####
**Evaluación de procedimientos**

En la evaluación de procedimientos se deben validar los siguientes aspectos fundamentales:
- El `rator` debe ser un `procVal`.
- El número de argumentos debe ser igual al número que espera el `procVal`.
- Los tipos de los argumentos deben coincidir con los tipos esperados por el procedimiento.

Al evaluar un procedimiento, por ejemplo `let f = proc (int x) x in (f 3)`, el resultado de la evaluación es el tipo de retorno del `proc`. En este caso, si `f` es de tipo `int -> int`, entonces `(f 3)` es de tipo `int`.

####
**Implementación de `type-of-expression` para aplicaciones**

Para evaluar aplicaciones, se modifican los ambientes iniciales dentro de `type-of-expression`. Se calcula el tipo del operador (`trator`) y los tipos de los operandos (`trands`), para posteriormente invocar `type-of-application`.

```racket
(define type-of-expression
  (lambda (exp tenv)
    ...
    (app-exp (rator rands)
      (let ((trator (type-of-expression rator tenv))
            (trands (map (lambda (x) (type-of-expression x tenv)) rands)))
        (type-of-application trator trands rator rands exp)))))
```

####
**Validación con `type-of-application`**

La función `type-of-application` verifica que el `trator` sea efectivamente un `proc-type`, compara las longitudes de las listas de argumentos y asegura la correspondencia de tipos mediante `check-equal-type!`.

```racket
(define type-of-application
  (lambda (trator trands rator rands exp)
    (cases type trator
      (proc-type (args-types result-type)
        (if (= (length args-types) (length trands))
            (begin
              (for-each check-equal-type! trands args-types rands)
              result-type)
            (eopl:error 'type-of-application
                        "Length of the args mismatch ~s ~s"
                        args-types trands)))
      (else
        (eopl:error 'type-of-expression
                    "Rator isn't a proc-type ~s is a type ~s"
                    rator
                    (type-to-external-form trator))))))
```

####
**Entornos iniciales y de tipos**

Se definen los entornos `env0` e `init-env` para la evaluación de valores, y el `init-tenv` para la verificación de tipos, asegurando que las variables y procedimientos tengan sus tipos correspondientes definidos.

```racket
(define env0
  (extend-env '(a b c)
              (list->vector '(4 5 6))
              (empty-env)))

(define init-env
  (extend-env '(x y z f)
              (list->vector (list 1 2 3
                                  (closure '(x y)
                                           (prim-exp (add-prim)
                                                     (list (var-exp 'x) (var-exp 'y)))
                                           env0)))
              env0))

(define init-tenv
  (extend-env '(x y z f)
              (list->vector (list int-type int-type int-type
                                  (proc-type (list int-type int-type int-type) int-type)))
              (extend-env '(a b c)
                          (list->vector (list int-type int-type int-type))
                          (empty-env))))
```

####
**Ejemplos de evaluación de tipos**

Utilizando `init-tenv`, se puede evaluar el tipo de una expresión dada. En el primer ejemplo se obtiene el tipo del procedimiento `f`, y en el segundo el tipo resultante de aplicar dicho procedimiento.

```racket
;; Obtención del tipo de f
(type-of-expression (cases program (parser "f") (a-program (exp) exp)) init-tenv)
;; Resultado: #(struct:proc-type (#(struct:atomic-type int) #(struct:atomic-type int) #(struct:atomic-type int)) #(struct:atomic-type int))

;; Obtención del tipo de la aplicación (f 1 2 3)
(type-of-expression (cases program (parser "(f 1 2 3)") (a-program (exp) exp)) init-tenv)
;; Resultado: #(struct:atomic-type int)
```
