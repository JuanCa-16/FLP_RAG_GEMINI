Vamos a una especificación léxica y gramatical. El intérprete nos proporciona un scanner y un parser para transformar el código en un Árbol de Sintaxis Abstracta (AST). Nuestra tarea siempre será trabajar el AST para retornar una respuesta al programador. Vamos a un prompt (REPL) que nos permitirá introducir el código y generaremos un ambiente para tener ligaduras, para lo cual usaremos datatypes.

###
El siguiente código define el lenguaje base (`lang eopl`) y presenta una definición de la gramática en formato BNF extendido como un comentario. Esto sirve como una guía para la estructura sintáctica de las expresiones que el intérprete procesará.

```scheme
#lang eopl
; Lenguaje Essentials of Programming Languages
; Definición de la gramática en comentario (BNF extendido)
#|
<expression> ::= <identifier>
                 var-exp(id)
             ::= <number>
                 lit-exp(dat)
             ::= "true"
                 true-exp()
             ::= "false"
                 false-exp()
             ::= <primitive> "(" <expression>* (,) ")"
                 prim-exp(prim, rands)
<primitive>  ::= "+" plus-prim()
             ::= "-" sub-prim()
             ::= "*" times-prim()
             ::= "/" div-prim()
             ::= "and" and-prim()
             ::= "or" or-prim()
             ::= ">" more-prim()
             ::= ">=" moreeq-prim()
             ::= "<" less-prim()
             ::= "<=" lesseq-prim()
|#
```

###
Esta sección define la especificación léxica, que es crucial para la primera fase de un intérprete: el escaneo. Aquí se definen los tipos de tokens (unidades mínimas con significado) y los patrones de expresiones regulares que los reconocen. Se incluyen reglas para comentarios (ignorados), espacios en blanco (ignorados), números (positivos y negativos) e identificadores.

```scheme
; Especificación léxica: define tokens y patrones regex
(define lexical
  '(
    (comment
     ("%" (arbno (not #\newline)))
     skip)
    ; Comentarios: % hasta newline (se ignoran)
    (whitespace
     (whitespace)
     skip)
    ; Espacios en blanco (se ignoran)
    (number
     (digit (arbno digit))
     number)
    ; Números positivos: dígitos
    (number
     ("-" digit (arbno digit))
     number)
    ; Números negativos: - seguido de dígitos
    (identifier
     (letter (arbno (or letter digit)))
     symbol)
    ; Identificadores: letra seguida de letras/dígitos
    ))
```

###
La gramática para el parser define las reglas sintácticas del lenguaje y especifica cómo construir el Árbol de Sintaxis Abstracta (AST) para cada constructo. Aquí se detalla la estructura de un programa (una expresión), los diferentes tipos de expresiones (variables, literales, booleanos, primitivas con argumentos) y las diversas operaciones primitivas soportadas, como aritméticas, lógicas y de comparación.

```scheme
; Gramática para el parser (producciones y constructores del AST)
(define grammar
  '(
    (program
     (expression)
     a-program)
    ; Programa: una expresión
    (expression
     (identifier)
     var-exp)
    ; Expresión: variable
    (expression
     (number)
     lit-exp)
    ; Expresión: literal numérico
    (expression
     ("true")
     true-exp)
    ; Expresión: booleano true
    (expression
     ("false")
     false-exp)
    ; Expresión: booleano false
    (expression
     (primitive "(" (separated-list expression ",") ")")
     prim-exp)
    ; Expresión primitiva con argumentos
    (primitive
     ("+")
     add-prim)
    ; Primitiva: suma
    (primitive
     ("-")
     sub-prim)
    ; Primitiva: resta
    (primitive
     ("*")
     prod-prim)
    ; Primitiva: multiplicación
    (primitive
     ("/")
     div-prim)
    ; Primitiva: división
    (primitive
     ("and")
     and-prim)
    ; Primitiva: and lógico
    (primitive
     ("or")
     or-prim)
    ; Primitiva: or lógico
    (primitive
     ("<")
     less-prim)
    ; Primitiva: menor que
    (primitive
     ("<=")
     lesseq-prim)
    ; Primitiva: menor o igual
    (primitive
     (">")
     more-prim)
    ; Primitiva: mayor que
    (primitive
     (">=")
     moreeq-prim)
    ; Primitiva: mayor o igual
    (primitive
     ("==")
     eq-prim)
    ; Primitiva: igualdad
    (primitive
     ("!=")
     neq-prim)
    ; Primitiva: desigualdad
    ))
```

###
Una vez definidas las especificaciones léxica y gramatical, se utilizan las funciones `sllgen:make-define-datatypes`, `sllgen:make-string-scanner` y `sllgen:make-string-parser` para generar automáticamente las estructuras de datos del AST y las funciones del scanner y el parser.

El `scanner` convierte una cadena de texto del programa en una lista de tokens, mientras que el `parser` toma esa lista de tokens (o la cadena directamente en este caso) y la transforma en un Árbol de Sintaxis Abstracta (AST) que representa la estructura del programa.

```scheme
;; Generar datatypes automáticamente a partir de lexical/grammar
(sllgen:make-define-datatypes lexical grammar)

; Scanner: convierte string en lista de tokens
; scanner -> string -> lista de tokens
(define scanner
  (lambda (program)
    ((sllgen:make-string-scanner lexical grammar) program)))

; Parser: convierte string en AST usando la gramática
; parser: string -> ast
(define parser
  (lambda (program)
    ((sllgen:make-string-parser lexical grammar) program)))
```

###
El intérprete define una estructura de datos `environment` para gestionar las ligaduras de variables. El `empty-env` representa un entorno sin variables, y `extend-env` permite añadir nuevas ligaduras (identificadores y sus valores) a un entorno existente. La función `value?` es un predicado simplificado que considera cualquier valor como válido.

La función `apply-env` es fundamental para la búsqueda de variables: dado un entorno y un identificador, busca el valor asociado a ese identificador, recorriendo recursivamente los entornos anidados desde el más reciente hasta el más antiguo. Si la variable no se encuentra, se genera un error.

```scheme
; Definición del entorno (environment) como estructura de datos
(define-datatype environment enviroment?
  (empty-env)
  ; Entorno vacío
  (extend-env
   ; Extender entorno con nuevos bindings
   (lid (list-of symbol?))
   ; Lista de identificadores
   (lval (list-of value?))
   ; Lista de valores
   (old-env enviroment?)))
; Entorno anterior

; Predicado value?: cualquier valor es válido (implementación simplificada)
(define value?
  (lambda (v)
    #t))

; Aplicar entorno: busca un símbolo en el entorno y retorna su valor
; apply-env: environment -> value
(define apply-env
  (lambda (env var)
    (cases environment env
      (empty-env ()
        (eopl:error "Variable not found" var))
      ; Error si no se encuentra
      (extend-env (lid lval old-env)
        (letrec ((search-var
                  ; Búsqueda recursiva en la lista de bindings
                  (lambda (lid lval)
                    (cond
                      [(null? lid)
                       (apply-env old-env var)]
                      ; Buscar en entorno anterior
                      [(equal? (car lid) var)
                       (car lval)]
                      ; Encontrado
                      [else
                       (search-var (cdr lid) (cdr lval))]))
                  ; Continuar búsqueda
                  ))
          (search-var lid lval))))))
```

###
`eval-program` es el punto de entrada principal para evaluar un programa completo. Toma el AST de un programa y llama a `eval-expression` para evaluar su expresión principal, utilizando un `init-env` predefinido. Este entorno inicial ya contiene algunas variables (`x`, `y`, `z`, `a`, `b`, `c`) con valores numéricos para facilitar las pruebas.

La función `eval-expression` es el corazón del intérprete, evaluando recursivamente los distintos tipos de expresiones definidas en la gramática. Para `var-exp`, busca la variable en el entorno; para `lit-exp`, `true-exp`, y `false-exp`, retorna directamente el valor literal o booleano. Para `prim-exp`, primero evalúa todos sus argumentos en el entorno actual y luego aplica la función `apply-primitive` con la primitiva y los valores resultantes.

```scheme
; Evaluar programa: punto de entrada, evalúa la expresión del programa
; eval-program: program -> value
(define eval-program
  (lambda (pgm)
    (cases program pgm
      (a-program (exp)
        (eval-expression exp init-env))
      ; Evalúa la expresión con entorno inicial
      )))

; Entorno inicial predefinido (variables x,y,z y a,b,c con valores numéricos)
(define init-env
  (extend-env
   '(x y z)
   '(1 2 3)
   (extend-env
    '(a b c)
    '(4 5 6)
    (empty-env))))

; Evaluar expresión: recursivamente evalúa según el tipo de expresión
; eval-expression: expression -> value
(define eval-expression
  (lambda (exp env)
    (cases expression exp
      (var-exp (id)
        (apply-env env id))
      ; Variable: busca en entorno
      (lit-exp (datum)
        datum)
      ; Literal: retorna el número
      (true-exp ()
        #t)
      ; True: retorna #t
      (false-exp ()
        #f)
      ; False: retorna #f
      (prim-exp (prim lrands)
        ; Primitiva: evalúa argumentos y aplica
        (let ((values
               (map
                (lambda (x)
                  (eval-expression x env))
                lrands)))
          (apply-primitive prim values)))
      (else 0))))
; Default (no debería ocurrir)
```

###
La función `operation` es una utilidad genérica que aplica una función binaria (`f`) de manera acumulativa a una lista de valores, comenzando con un valor inicial (`acc`). Esta función es utilizada por `apply-primitive` para implementar operaciones como la suma, el producto, el AND y el OR, que pueden tomar múltiples argumentos.

`apply-primitive` es responsable de ejecutar la lógica específica para cada operación primitiva. Utiliza `cases primitive` para distinguir entre las diferentes primitivas. Las operaciones aritméticas como `add-prim`, `prod-prim` usan `operation` para sumar o multiplicar todos los valores. `sub-prim` y `div-prim` tratan el primer argumento de manera especial y los restantes de forma acumulativa. Las operaciones booleanas `and-prim` y `or-prim` también usan `operation` con los valores iniciales apropiados (`#t` y `#f`). Las comparaciones (`less-prim`, `eq-prim`, etc.) esperan exactamente dos argumentos y los comparan directamente.

```scheme
; Operación genérica: aplica una función binaria acumulativa a una lista
; operation: (T,T)->T, lista de valores, T -> value
(define operation
  (lambda (f lst acc)
    (cond
      [(null? lst)
       acc]
      ; Caso base: retorna acumulador
      [else
       (operation f (cdr lst) (f (car lst) acc))]
      ; Aplica f y acumula
      )))

; Aplicar primitiva: ejecuta la operación correspondiente sobre los valores
; apply-primitive: primitive, lista valores -> value
(define apply-primitive
  (lambda (prim values)
    (cases primitive prim
      (add-prim ()
        (operation + values 0))
      ; Suma acumulativa
      (sub-prim ()
        (- (car values) (operation + (cdr values) 0)))
      ; Resta (first - sum(rest))
      (prod-prim ()
        (operation * values 1))
      ; Producto acumulativo
      (div-prim ()
        (/ (car values) (operation * (cdr values) 1)))
      ; División (first / product(rest))
      (and-prim ()
        (operation (lambda (a b) (and a b)) values #t))
      ; AND acumulativo
      (or-prim ()
        (operation (lambda (a b) (or a b)) values #f))
      ; OR acumulativo
      (less-prim ()
        (< (car values) (cadr values)))
      ; < (2 args)
      (lesseq-prim ()
        (<= (car values) (cadr values)))
      ; <= (2 args)
      (more-prim ()
        (> (car values) (cadr values)))
      ; > (2 args)
      (moreeq-prim ()
        (>= (car values) (cadr values)))
      ; >= (2 args)
      (eq-prim ()
        (= (car values) (cadr values)))
      ; == numérico (2 args)
      (neq-prim ()
        (not (= (car values) (cadr values))))
      ; != numérico (2 args)
      )))
```

###
El intérprete interactivo se implementa utilizando `sllgen:make-rep-loop`, que crea un bucle de lectura-evaluación-impresión (REPL). Este REPL utiliza "-->" como prompt, `eval-program` para la evaluación y un parser basado en stream para procesar la entrada. Al llamar a `(interpreter)`, se inicia este bucle interactivo, permitiendo al usuario introducir expresiones y ver sus resultados directamente.

El análisis del intérprete destaca que los valores denotados y expresados son números y booleanos de Scheme. La estructura del intérprete se basa en `eval-program` como punto de entrada y `eval-expression` para la evaluación recursiva de los distintos tipos de expresiones del AST. El manejo de entornos utiliza `extend-env` para ligaduras con alcance léxico y `apply-env` para la búsqueda de variables. Las primitivas incluyen operaciones aritméticas, lógicas y de comparación, muchas de las cuales utilizan la función `operation` para manejar múltiples argumentos.

```scheme
; Intérprete interactivo: loop de lectura-evaluación-impresión
(define interpreter
  (sllgen:make-rep-loop
   "-->"
   ; Prompt
   eval-program
   ; Función de evaluación
   (sllgen:make-stream-parser lexical grammar)
   ; Parser para input stream
   ))

; Iniciar el intérprete
(interpreter)
```

###
Entre las limitaciones del intérprete actual se encuentran: el tipado débil, que no realiza verificación de tipos; la aridad fija de las comparaciones, que solo funcionan con dos argumentos a pesar de que la gramática permitiría más; y el comportamiento no estándar de la división y resta con múltiples argumentos (por ejemplo, `(- a b c)` se evalúa como `a - (b + c)` y `(/ a b c)` como `a / (b * c)`). Además, no hay restricciones para mezclar booleanos y números en operaciones inapropiadas, como `(+ true 5)`.

El intérprete aprovecha las convenciones de EOPL, como el uso de `cases` para el pattern matching en ASTs, la generación automática de datatypes con `sllgen:make-define-datatypes` y el manejo de errores con `eopl:error`. Para futuras mejoras se sugiere añadir verificación de tipos, manejar la aridad variable en las comparaciones y definir un valor denotado explícito con una estructura como `(define-datatype expval ...)`.

###
Para probar el intérprete, es necesario comentar la línea `(interpreter)` para evitar que se levante el REPL interactivo. Luego, se pueden ejecutar llamadas directas a `eval-program` con expresiones parseadas.

```scheme
; Debe comentar la ejecución del interprete, para evitar que el REPL se levante (prompt)
; (interpreter)

(eval-program (parser "x"))
; 1
(eval-program (parser "y"))
; 2
(eval-program (parser "+(x,y)"))
; 3
```
