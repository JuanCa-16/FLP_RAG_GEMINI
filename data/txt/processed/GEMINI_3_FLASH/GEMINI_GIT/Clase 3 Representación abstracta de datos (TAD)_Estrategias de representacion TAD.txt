####
**Estrategias de implementación de TADs en EOPL**
Un `Tipo Abstracto de Dato` (`TAD`) es una especificación matemática de un conjunto de datos y las operaciones definidas sobre ellos, independiente de su implementación concreta en memoria. En `Essentials of Programming Languages` (`EOPL`), los `TADs` enfatizan la separación entre la interfaz (qué hace el tipo) y la implementación (cómo lo hace).

**Contraste con tipos primitivos**
Un tipo primitivo como `int` representa una dependencia directa del hardware (por ejemplo, 32 bits).
- Ventajas: Operaciones eficientes a nivel de `CPU` y soporte nativo del lenguaje.
- Desventajas: Rango limitado (de -2^31 a 2^31-1), problemas de desbordamiento y comportamiento dependiente de la plataforma.

**Abstracción en lenguajes de alto nivel**
En lenguajes de alto nivel, los objetos tienen un comportamiento definido donde las operaciones tienen una implementación interna oculta.
```python
a = 3 # Objeto con comportamiento definido
```
```racket
(define x 10) # Representación abstracta
```

####
**Representación recursiva de números naturales**
Basado en los axiomas de Peano, los números naturales se pueden definir de la siguiente forma:
- 0 pertenece a N
- Si n pertenece a N, entonces n + 1 pertenece a N

Esta definición matemática permite implementar el `TAD` de números naturales mediante diversas estrategias, asegurando que la lógica de las operaciones sea consistente independientemente de cómo se almacenen los datos.

####
**Implementación de números naturales usando números nativos**
Esta implementación utiliza los enteros del lenguaje anfitrión para definir las operaciones del `TAD`.

```racket
#lang eopl
;; === IMPLEMENTACIÓN (OCULTA) ===
(define zero 0)

(define isZero?
  (lambda (n)
    (eqv? n zero)))

(define succ
  (lambda (n)
    (+ n 1)))

(define pred
  (lambda (n)
    (- n 1)))

;; === INTERFAZ (VISIBLE) ===
(define cinco (succ (succ (succ (succ (succ zero))))))

(define suma
  (lambda (a b)
    (cond
      [(isZero? b) a]
      [else (suma (succ a) (pred b))])))

(define multiplicacion
  (lambda (a b)
    (cond
      [(isZero? b) zero]
      [(eqv? (succ zero) b) a]
      [else (suma a (multiplicacion a (pred b)))])))

(define diez (suma cinco cinco))
(define cincuenta (multiplicacion cinco diez))
```
En este esquema, aunque se usan números nativos, el programador interactúa con el tipo a través de `succ`, `pred` e `isZero?`.

####
**Implementación de números naturales usando listas (representación unaria)**
En esta implementación alternativa, el número se representa mediante la longitud de una lista.

```racket
#lang eopl
;; === IMPLEMENTACIÓN ALTERNATIVA ===
(define zero '())

(define isZero?
  (lambda (n)
    (eqv? n zero)))

(define succ
  (lambda (n)
    (cons #t n)))

(define pred
  (lambda (n)
    (cdr n)))

;; === MISMA INTERFAZ ===
;; El código del programador permanece IDÉNTICO
(define cinco (succ (succ (succ (succ (succ zero))))))
```
A pesar de que la estructura interna cambió de un entero a una lista, las funciones de alto nivel como `suma` y `multiplicacion` funcionan sin cambios porque dependen de la interfaz abstracta.

####
**Implementación de números naturales en base 16 (Bignum)**
Esta estrategia es una representación posicional eficiente para manejar números grandes mediante listas de dígitos.

```racket
#lang eopl
;; === IMPLEMENTACIÓN EFICIENTE ===
(define zero '())
(define base 16)

(define isZero?
  (lambda (n)
    (equal? n zero)))

(define succ
  (lambda (n)
    (cond
      [(isZero? n) '(1)]
      [(equal? (- base 1) (car n))
       (cons 0 (succ (cdr n)))]
      [else (cons (+ (car n) 1) (cdr n))])))

(define pred
  (lambda (n)
    (cond
      [(equal? n (list 1)) zero]
      [(equal? 0 (car n))
       (cons (- base 1) (pred (cdr n)))]
      [else (cons (- (car n) 1) (cdr n))])))
```
Esta implementación gestiona acarreos y préstamos de forma interna. El usuario final sigue utilizando las mismas operaciones abstractas sin notar el cambio en la eficiencia o en la estructura de datos subyacente.

####
**Análisis de la abstracción de datos**
Los puntos fundamentales de los `TADs` según `EOPL` incluyen:
- **Ocultamiento de información**: La implementación concreta está encapsulada y oculta al usuario.
- **Invariantes preservadas**: Las operaciones mantienen la validez del tipo de dato.
- **Sustituibilidad**: Diferentes implementaciones pueden satisfacer la misma interfaz.
- **Independencia de representación**: El comportamiento lógico no depende de la representación física en memoria.

Esto permite realizar optimizaciones transparentes y corregir errores en la implementación sin afectar el código cliente que ya utiliza el `TAD`.

####
**Componentes fundamentales de un TAD**
Todo `Tipo Abstracto de Dato` requiere los siguientes componentes para su definición funcional:
- **Procedimientos constructores**: Encargados de crear instancias válidas del `TAD` (ej. `zero`, `succ`, `var-exp`).
- **Procedimientos observadores**:
    - **Predicados**: Determinan si un dato pertenece a una variante específica del `TAD` (ej. `isZero?`, `lambda-exp?`).
    - **Extractores**: Permiten obtener componentes específicos de los datos (ej. extraer el cuerpo de una función `lambda`).

####
**Estrategia de representación basada en listas**
Esta estrategia utiliza listas de `Racket` donde el primer elemento funciona como una etiqueta (`tag`) para identificar el tipo de expresión.

```racket
#lang eopl
;; === CONSTRUCTORES (Interfaz) ===
(define var-exp
  (lambda (id)
    (list 'var-exp id)))

(define lambda-exp
  (lambda (id exp)
    (list 'lambda-exp id exp)))

(define app-exp
  (lambda (rator rand)
    (list 'app-exp rator rand)))

;; === OBSERVADORES (Predicados) ===
(define var-exp?
  (lambda (exp)
    (equal? (car exp) 'var-exp)))

(define lambda-exp?
  (lambda (exp)
    (equal? (car exp) 'lambda-exp)))

(define app-exp?
  (lambda (exp)
    (equal? (car exp) 'app-exp)))
```

####
**Extractores y aplicación en la representación basada en listas**
Los extractores permiten recuperar la información almacenada en las listas siguiendo la estructura definida por los constructores.

```racket
#lang eopl
;; === OBSERVADORES (Extractores) ===
(define var-exp->id (lambda (exp) (cadr exp)))
(define lambda-exp->id (lambda (exp) (cadr exp)))
(define lambda-exp->exp (lambda (exp) (caddr exp)))
(define app-exp->rator (lambda (exp) (cadr exp)))
(define app-exp->rand (lambda (exp) (caddr exp)))

;; === CÓDIGO DEL PROGRAMADOR ===
(define occurs-free?
  (lambda (exp v)
    (cond
      [(var-exp? exp) (equal? v (var-exp->id exp))]
      [(lambda-exp? exp)
       (and (not (equal? v (lambda-exp->id exp)))
            (occurs-free? (lambda-exp->exp exp) v))]
      [(app-exp? exp)
       (or (occurs-free? (app-exp->rator exp) v)
           (occurs-free? (app-exp->rand exp) v))])))
```
El procedimiento `occurs-free?` demuestra que se puede implementar lógica compleja utilizando solo la interfaz del `TAD`, sin acceder directamente a la estructura de la lista.

####
**Estrategia de representación basada en procedimientos**
En esta estrategia, los datos no se guardan en listas, sino que se encapsulan dentro de `closures` (procedimientos) que responden a mensajes o selectores numéricos.

```racket
#lang eopl
;; === CONSTRUCTORES (Implementación con closures) ===
(define var-exp
  (lambda (id)
    (lambda (s)
      (cond
        [(= s 0) 'var-exp]
        [(= s 1) id]
        [else (eopl:error "Error en var-exp")]))))

(define lambda-exp
  (lambda (id exp)
    (lambda (s)
      (cond
        [(= s 0) 'lambda-exp]
        [(= s 1) id]
        [(= s 2) exp]
        [else (eopl:error "Error en lambda-exp")]))))

(define app-exp
  (lambda (rator rand)
    (lambda (s)
      (cond
        [(= s 0) 'app-exp]
        [(= s 1) rator]
        [(= s 2) rand]
        [else (eopl:error "Error en app-exp")]))))
```

####
**Observadores en la representación basada en procedimientos**
Para esta estrategia, los observadores invocan al dato (que es un procedimiento) pasando el índice del componente que se desea recuperar.

```racket
#lang eopl
;; === OBSERVADORES ===
;; Predicados
(define var-exp? (lambda (exp) (equal? (exp 0) 'var-exp)))
(define lambda-exp? (lambda (exp) (equal? (exp 0) 'lambda-exp)))
(define app-exp? (lambda (exp) (equal? (exp 0) 'app-exp)))

;; Extractores
(define var-exp->id (lambda (exp) (exp 1)))
(define lambda-exp->id (lambda (exp) (exp 1)))
(define lambda-exp->exp (lambda (exp) (exp 2)))
(define app-exp->rator (lambda (exp) (exp 1)))
(define app-exp->rand (lambda (exp) (exp 2)))
```
Aunque la implementación cambió internamente a `procedimientos`, la función `occurs-free?` y cualquier otro código cliente permanecen idénticos, validando el principio de independencia de representación.

####
**Comparación de estrategias de representación**
Ambas estrategias implementan el mismo `TAD` con diferentes compromisos:
- **Listas**: Es más sencilla de implementar y depurar, con menor sobrecarga computacional, pero ofrece menos encapsulación.
- **Procedimientos**: Proporciona una encapsulación fuerte y validación de acceso incorporada, aunque conlleva una mayor carga computacional debido al uso intensivo de `closures`.

La clave del diseño de sistemas elegantes es asegurar que la interfaz abstracta permanezca constante para que los cambios en la implementación no afecten al resto del software.
