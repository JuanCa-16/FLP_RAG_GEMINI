####
**Definición y funcionamiento de los lenguajes compilados**

Los `lenguajes compilados` son aquellos que transforman el código fuente en `código máquina`, el cual se ejecuta directamente sobre la `CPU`. Alternativamente, pueden generar un `código binario` diseñado para ser ejecutado por un programa específico, como una `Máquina Virtual del Lenguaje`. Este código usualmente atraviesa varias etapas de `optimización` con el objetivo de lograr una ejecución lo más rápida posible.

```mermaid
graph TD
    A["Código fuente"] --> |Parser| B["AST"]
    B --> |Traducción| C["Código binario"]
```

En el proceso de traducción, el `AST` (Abstract Syntax Tree) es transformado a `código binario` para la `CPU` o para un programa que lo ejecute, como es el caso de la `Máquina Virtual de Java` (JVM). Al no poseer intermediarios directos con la `CPU` durante su ejecución final, estos lenguajes se ejecutan más rápidamente y su sintaxis suele ser más cercana al `lenguaje máquina`, como ocurre en `C++`.

####
**Definición y comportamiento de los lenguajes interpretados**

Un `lenguaje interpretado` es aquel cuyas instrucciones son leídas y procesadas por otro programa denominado `intérprete`. A diferencia de lenguajes como `Java`, donde un error de sintaxis impide cualquier ejecución, en lenguajes como `Python` el programa se ejecuta secuencialmente hasta encontrar el error.

El `intérprete` lee las instrucciones línea por línea. Esto implica que el código fuente es traducido por otro programa que envía las instrucciones a la `CPU`, actuando como un intermediario.

```mermaid
graph TD
    A["Código fuente"] --> |Frontend| B["AST"]
    B --> |Interpretado| C["Salida"]
```

####
**El rol del Frontend y el Parser en la interpretación**

El proceso de generación del `AST` (Abstract Syntax Tree) es realizado por un componente denominado `Frontend`.

```mermaid
graph TD
    A["Parser"] --> |AST| B["Interpretación"]
```

El `Parser` tiene la capacidad de enviar el `AST` directamente al `intérprete` o generar código en un `lenguaje intermedio`. Este lenguaje intermedio es el que finalmente se interpreta, lo cual permite realizar diversas `optimizaciones` en el flujo de ejecución.

####
**Comparativa técnica entre lenguajes compilados e interpretados**

| Aspecto | Lenguajes Compilados | Lenguajes Interpretados |
| :--- | :--- | :--- |
| **Ejecución** | Se ejecuta directamente en la `CPU` o en una máquina virtual | Requiere un `intérprete` que lee y ejecuta línea por línea |
| **Velocidad** | Mayor rendimiento (sin intermediarios) | Menor rendimiento (existe `overhead` del intérprete) |
| **Detección de errores** | Errores detectados en `tiempo de compilación` (antes de ejecutar) | Errores detectados en `tiempo de ejecución` (hasta la línea fallida) |
| **Portabilidad** | Depende de la arquitectura (requiere recompilación) | Más portables (el mismo código fuente corre en múltiples plataformas con intérprete) |
| **Optimización** | Optimizaciones agresivas en tiempo de compilación | Optimizaciones limitadas (en tiempo de ejecución o con `JIT`) |
| **Flexibilidad** | Menos flexible (cambios requieren recompilación) | Más flexible (cambios en código fuente efectivos inmediatamente) |
| **Ejemplos típicos** | `C`, `C++`, `Rust`, `Go` | `Python`, `JavaScript`, `Ruby`, `PHP` |
| **Proceso de traducción** | Código fuente → `AST` → Código binario | Código fuente → `AST` → Interpretación línea por línea |
| **Uso de recursos** | Menor uso de memoria en ejecución (estático) | Mayor uso de memoria (por el intérprete y entorno `runtime`) |
| **Depuración** | Más difícil (depuración a nivel de binario) | Más fácil (depuración a nivel de código fuente) |

####
**Consideraciones sobre la elección y modelos híbridos**

La elección entre un modelo compilado o interpretado depende del equilibrio necesario entre rendimiento, portabilidad y flexibilidad. Es importante considerar los siguientes puntos clave:

- Los `lenguajes compilados` generan `código binario` optimizado para la `CPU` o una máquina virtual (ejemplo: `JVM` para `Java`).
- Los `lenguajes interpretados` dependen de un `intérprete` que traduce y ejecuta instrucciones en tiempo real.
- Muchos lenguajes modernos utilizan enfoques híbridos; por ejemplo, `Java` compila a `bytecode`, el cual es posteriormente interpretado o procesado mediante un compilador `JIT` (Just-In-Time).
