####
**Tipo Abstracto de Dato (TAD) en EOPL**

Un `Tipo Abstracto de Dato` (TAD) es una especificación matemática de un conjunto de datos y las operaciones definidas sobre ellos, independiente de su implementación concreta en memoria. En *Essentials of Programming Languages* (EOPL), los TADs enfatizan la separación fundamental entre:

- **Interfaz**: Define qué hace el tipo (la especificación).
- **Implementación**: Define cómo lo hace (la representación interna).

A diferencia de los tipos primitivos (como un `int` de 32 bits), que dependen del hardware y presentan limitaciones como rangos finitos o desbordamientos, los TADs permiten abstraer el comportamiento del objeto, manteniendo la lógica de las operaciones oculta del programador.

####
**Representación recursiva de números naturales**

Basado en los axiomas de Peano, los números naturales se definen como:
- $\lceil 0 \rceil \in \mathbb{N}$
- $n \in \mathbb{N} \rightarrow n+1 \in \mathbb{N}$

Esta representación permite implementar números utilizando diferentes estrategias sin alterar la interfaz de uso.

**Implementación 1: Usando números nativos**

```racket
#lang eopl
;; === IMPLEMENTACIÓN (OCULTA) ===
(define zero 0)
(define isZero? (lambda (n) (eqv? n zero)))
(define succ (lambda (n) (+ n 1)))
(define pred (lambda (n) (- n 1)))

;; === INTERFAZ (VISIBLE) ===
(define cinco (succ (succ (succ (succ (succ zero))))))
(define suma (lambda (a b)
               (cond [(isZero? b) a]
                     [else (suma (succ a) (pred b))])))
(define diez (suma cinco cinco))
```

Esta implementación utiliza las operaciones aritméticas nativas del lenguaje para definir los números naturales mediante un esquema de `sucesor` y `predecesor`.

####
**Implementaciones alternativas de números naturales**

Es posible cambiar la representación interna del TAD sin modificar el código del programador (*código cliente*).

**Implementación 2: Usando listas (representación unaria)**
```racket
#lang eopl
(define zero '())
(define isZero? (lambda (n) (eqv? n zero)))
(define succ (lambda (n) (cons #T n)))
(define pred (lambda (n) (cdr n)))
```

**Implementación 3: Representación en base 16 (bignum)**
```racket
#lang eopl
(define zero '())
(define base 16)
(define isZero? (lambda (n) (equal? n zero)))
(define succ (lambda (n)
               (cond [(isZero? n) '(1)]
                     [(equal? (- base 1) (car n)) (cons 0 (succ (cdr n)))]
                     [else (cons (+ (car n) 1) (cdr n))])))
```
Estas estrategias demuestran el principio de **independencia de representación**, donde la lógica del cliente (operaciones como `suma` o `multiplicacion`) funciona de manera idéntica independientemente de si los datos se almacenan como números, listas o representaciones posicionales.

####
**Componentes fundamentales de un TAD**

Todo `TAD` se compone de los siguientes elementos necesarios para su operatividad:

- **Procedimientos constructores**: Se encargan de crear instancias válidas del TAD.
- **Procedimientos observadores**:
    - **Predicados**: Determinan si un dato pertenece a una variante específica del TAD.
    - **Extractores**: Permiten obtener componentes específicos almacenados dentro de los datos.

####
**Estrategia de representación basada en listas**

En esta estrategia, el TAD se representa mediante una lista donde el primer elemento funciona como una etiqueta (`tag`) que identifica el tipo de dato.

```racket
#lang eopl
;; === CONSTRUCTORES ===
(define var-exp (lambda (id) (list 'var-exp id)))
(define lambda-exp (lambda (id exp) (list 'lambda-exp id exp)))
(define app-exp (lambda (rator rand) (list 'app-exp rator rand)))

;; === OBSERVADORES (Predicados) ===
(define var-exp? (lambda (exp) (equal? (car exp) 'var-exp)))

;; === EXTRACTORES ===
(define var-exp->id (lambda (exp) (cadr exp)))
```
Esta forma facilita la inspección de la estructura de datos, aunque el programador debe respetar la interfaz de extractores para acceder a los valores.

####
**Estrategia de representación basada en procedimientos**

Esta técnica utiliza `closures` para encapsular los datos, donde el constructor devuelve una función que actúa como un selector de campos.

```racket
#lang eopl
(define var-exp (lambda (id)
                  (lambda (s)
                    (cond [(= s 0) 'var-exp]
                          [(= s 1) id]
                          [else (eopl:error "Error en var-exp")]))))

(define var-exp? (lambda (exp) (equal? (exp 0) 'var-exp)))
(define var-exp->id (lambda (exp) (exp 1)))
```
**Análisis comparativo**:
- **Listas**: Ofrecen simplicidad y menor consumo de memoria.
- **Procedimientos**: Proporcionan una encapsulación más fuerte, validación de acceso y mayor flexibilidad, a costa de un mayor *overhead* computacional. 

Ambas estrategias cumplen el principio fundamental de que el **código cliente** permanece inalterado ante cambios en la implementación interna.
