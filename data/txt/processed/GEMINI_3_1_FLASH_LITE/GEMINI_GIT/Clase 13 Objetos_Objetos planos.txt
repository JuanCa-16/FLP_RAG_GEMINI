####
**Representación de objetos planos**

Los objetos planos, utilizados por lenguajes como `Java`, se implementan mediante un vector de campos. En esta estructura, las primeras posiciones del vector corresponden a los campos de las clases de mayor jerarquía, mientras que las últimas posiciones corresponden a las clases de menor jerarquía.

```text
class c1 extends object
  field x
  field y
  method initialize () 1

class c2 extends c1
  field z
  field w
  method initialize () 1
```

Cuando instanciamos estos objetos mediante `let o1 = new c1 ()` y `o2 = new c2 ()`, la representación interna es la siguiente:

- Para `o1`: `# (struct:an-object c1 # (0 0))`
- Para `o2`: `# (struct:an-object c2 # (0 0 0 0))`

En `o2`, el primer elemento del vector identifica la clase `c2` y los siguientes valores representan los campos heredados y propios. Se asume `object` como una lista vacía.

####
**Ejemplo de jerarquía con tres niveles**

Considerando una jerarquía de tres clases, la estructura de objetos se expande acumulando los campos de todas las clases superiores en el vector.

```text
class c1 extends object
  field x
  field y
  method initialize () 1

class c2 extends c1
  field z
  field w
  method initialize () 1

class c3 extends c2
  field a
  field b
  method initialize () 1
```

Al instanciar `let o3 = new c3 ()`, la representación resultante es:
`# (struct:an-object c3 # (0 0 0 0 0 0))`

Esta estructura se visualiza de la siguiente manera:

```mermaid
graph TD
    A["C3 (0 0 0 0 0 0)"]
```

Los primeros elementos del vector corresponden a la clase de mayor jerarquía (`c1`) y los últimos a la de menor jerarquía (`c3`).

####
**Ejemplo de ejecución con múltiples métodos**

A continuación se presenta un caso de uso con herencia y sobreescritura de métodos para observar el cambio en la representación interna de los campos:

```text
class p1 extends object
  field a
  field b
  method initialize (x, y) begin set a = x; set b = y end
  method sum (x) begin set a = +(a, 1); +(a, b, x, send self m1()) end
  method m1 () 1

class p2 extends p1
  field c
  field d
  field e
  method initialize (x, y, z) begin super initialize (x, y); set c = +(x, 1); set d = +(y, 2); set e = z end
  method sum (s) +(c, d, e, send self m1())

class p3 extends p2
  field f
  field g
  method initialize (u) begin set f = u; set g = +(u, 1); super initialize (u, +(u, 1), +(u, 2)) end
  method sum (a) +(a, super sum (+ (a, 2)))
  method m1 () 10
```

Al ejecutar `list (send o1 sum(2), send o2 sum(3), send o3 sum(4))` con las instancias iniciales, el resultado es `(9 16 38)`.

####
**Visualización de los estados de los objetos**

Tras la inicialización de los objetos, sus representaciones internas en memoria son:

Para `o1 = new p1(2,3)`:
```mermaid
graph TD
    A["p1 (2,3)"]
```
`# (struct:an-object p1 # (2 3))`

Para `o2 = new p2(3,4,5)`:
```mermaid
graph TD
    A["p2 (3,4,4,6,5)"]
```
`# (struct:an-object p2 # (3 4 4 6 5))`

Para `o3 = new p3(6)`:
```mermaid
graph TD
    A["p3 (6,7,7,9,8,6,7)"]
```
`# (struct:an-object p3 # (6 7 7 9 8 6 7))`

En `o3`, los valores `6,7` pertenecen a `p1`, `7,9,8` a `p2`, y `6,7` a `p3`.

####
**Diagrama de ambientes para la invocación de métodos**

La invocación de métodos sigue las reglas de encapsulación: cada llamado es independiente, a menos que un método modifique los campos del objeto, lo cual altera su estado persistente.

Para `send o1 sum(2)`, el ambiente se organiza así:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1 (a,b: 2,3)"]
    C["envsum (x:2, super:object, self:o1)"]
    A --> B
    B --> C
```

Al ejecutar `set a = +(a, 1)`, el valor de `a` en `env1` cambia a `3`, actualizando la representación del objeto a `p1 (3,3)`. Al completar la evaluación de `+(3,3,2, send o1 m1())`, se invoca `m1` y se obtiene `9`.

####
**Invocación de métodos en objetos con herencia (o2 y o3)**

En el caso de `send o2 sum(3)`, se crean ambientes vinculados que permiten acceder a los campos de `p1` y `p2`:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1 (a,b: 3,4)"]
    C["env2 (c,d,e: 4,6,5)"]
    D["envsum (s:3, super:p1, self:o2)"]
    A --> B
    B --> C
    C --> D
```

Para `send o3 sum(4)`, la cadena de ambientes es más profunda debido a los tres niveles de herencia:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1 (a,b: 6,7)"]
    C["env2 (c,d,e: 7,9,8)"]
    D["env3 (f,g: 6,7)"]
    E["envsum (a:4, super:p2, self:o3)"]
    A --> B
    B --> C
    C --> D
    D --> E
```

Cuando se ejecuta `super sum`, se crea un nuevo ambiente que apunta a la clase padre (`p2`) con el estado actual de los campos, permitiendo la resolución correcta de los métodos heredados.
