####
**Definición de Tipo Abstracto de Dato (TAD)**

Un `Tipo Abstracto de Dato` (TAD) es una especificación matemática de un conjunto de datos y las operaciones definidas sobre ellos, la cual es independiente de su implementación concreta en memoria. En el contexto de *Essentials of Programming Languages* (`EOPL`), los TADs enfatizan la separación fundamental entre la `interfaz` (qué hace el tipo) y la `implementación` (cómo lo hace).

####
**Contraste entre tipos primitivos y TADs**

A diferencia de los tipos primitivos del lenguaje, que dependen directamente del hardware, los TADs ofrecen un nivel superior de abstracción.

Ejemplo de tipo primitivo (dependiente de hardware):
```c
int pollito = 8;
// Representación: 32 bits (000...1000)
```

**Ventajas de los tipos primitivos:**
- Operaciones a nivel de CPU (eficientes).
- Soporte nativo del lenguaje.

**Desventajas de los tipos primitivos:**
- Rango limitado ($-2^{31}$ a $2^{31}-1$).
- Problemas de desbordamiento.
- Comportamiento dependiente de la plataforma.

En lenguajes de alto nivel, la abstracción permite manipular objetos con comportamiento definido mientras la implementación interna permanece oculta.

####
**Representación recursiva de números naturales**

La definición de los números naturales se basa en los axiomas de Peano:
1. $\lceil 0 \rceil \in \mathbb{N}$
2. $n \in \mathbb{N} \rightarrow n+1 \in \mathbb{N}$

####
**Implementación de TAD: Números nativos**

Esta implementación utiliza las capacidades aritméticas del lenguaje como base para el TAD.

```racket
#lang eopl

;; === IMPLEMENTACIÓN (OCULTA) ===
(define zero 0)
(define (isZero? n) (eqv? n zero))
(define (succ n) (+ n 1))
(define (pred n) (- n 1))

;; === INTERFAZ (VISIBLE) ===
(define cinco (succ (succ (succ (succ (succ zero))))))

(define (suma a b)
  (cond [(isZero? b) a]
        [else (suma (succ a) (pred b))]))

(define diez (suma cinco cinco))

(define (multiplicacion a b)
  (cond [(isZero? b) zero]
        [(eqv? (succ zero) b) a]
        [else (suma a (multiplicacion a (pred b)))]))

(define cincuenta (multiplicacion cinco diez))
```

####
**Implementación de TAD: Representación unaria (listas)**

Esta alternativa emplea una estructura de datos basada en listas para representar los números, manteniendo la misma interfaz de usuario que la implementación anterior.

```racket
#lang eopl

;; === IMPLEMENTACIÓN ALTERNATIVA ===
(define zero '())
(define (isZero? n) (eqv? n zero))
(define (succ n) (cons #t n))
(define (pred n) (cdr n))

;; === MISMA INTERFAZ ===
(define cinco (succ (succ (succ (succ (succ zero))))))
```

####
**Implementación de TAD: Base 16 (bignum)**

Esta implementación es más eficiente al utilizar una representación posicional, demostrando que el código cliente no requiere cambios cuando la implementación interna es optimizada.

```racket
#lang eopl

;; === IMPLEMENTACIÓN EFICIENTE ===
(define zero '())
(define base 16)
(define (isZero? n) (equal? n zero))

(define (succ n)
  (cond [(isZero? n) '(1)]
        [(equal? (- base 1) (car n)) (cons 0 (succ (cdr n)))]
        [else (cons (+ (car n) 1) (cdr n))]))

(define (pred n)
  (cond [(equal? n (list 1)) zero]
        [(equal? 0 (car n)) (cons (- base 1) (pred (cdr n)))]
        [else (cons (- (car n) 1) (cdr n))]))
```

####
**Análisis de la abstracción en EOPL**

Los TADs proporcionan ventajas fundamentales para el diseño de software:
- `Ocultamiento de información`: La implementación interna está encapsulada.
- `Invariantes preservadas`: Las operaciones garantizan que el tipo mantenga su validez.
- `Sustituibilidad`: Diversas implementaciones pueden satisfacer la misma interfaz.
- `Independencia de representación`: El comportamiento lógico es independiente de la física.

**Componentes clave del diseño:**
- `Constructores`: `zero` y `succ` crean instancias válidas del tipo.
- `Observadores`: `isZero?` examina el valor sin exponer cómo está almacenado.
- `Operaciones`: Definen el comportamiento abstracto a través de `recursión estructural`.
- `Pattern matching`: Utiliza la estructura del tipo para definir el funcionamiento de los métodos.

####
**Conclusión sobre la abstracción de datos**

La abstracción de datos permite definir tipos exclusivamente mediante su comportamiento (interfaz). Esto permite realizar optimizaciones transparentes, corregir errores en la implementación sin afectar al código cliente y adaptar el software a distintos requisitos de rendimiento o memoria. Encapsular decisiones de implementación es una práctica esencial para construir sistemas elegantes, mantenibles y resistentes a cambios.
