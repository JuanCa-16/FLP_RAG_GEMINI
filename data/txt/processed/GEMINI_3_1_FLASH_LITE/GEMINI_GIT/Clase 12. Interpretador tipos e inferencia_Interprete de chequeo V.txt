####
**Tratamiento de primitivas**

Las `primitivas` se manejan como procedimientos clasificados según su tipo. Se definen principalmente tres categorías:
- `Primitivas numéricas`: Ingresan `int` y emiten `int`.
- `Primitivas relacionales`: Ingresan dos `int` y emiten `bool`.
- `Primitivas booleanas`: Ingresan `bool` y emiten `bool`.

El siguiente código implementa el chequeo de tipos para aplicaciones de primitivas en `Racket`:

```racket
(define type-of-expression
  (lambda (exp tenv)
    (cases expression exp
      (prim-exp (prim rands)
        (type-of-application
          (type-of-primitive prim rands)
          (map (lambda (x) (type-of-expression x tenv)) rands)))))))
```

Esta implementación trata la primitiva como una aplicación de procedimiento, calculando los tipos de los argumentos mediante `map` y validándolos con `type-of-primitive`.

####
**Implementación de `type-of-primitive`**

Esta función determina el tipo de retorno y los requisitos de argumentos para cada primitiva soportada, utilizando un `cases` sobre el tipo `primitive`:

```racket
(define type-of-primitive
  (lambda (prim rands)
    (cases primitive prim
      ;; Primitivas aritméticas: toman N enteros y retornan un entero
      (add-prim () (proc-type (map (lambda (x) int-type) rands) int-type))
      (sub-prim () (proc-type (map (lambda (x) int-type) rands) int-type))
      (prod-prim () (proc-type (map (lambda (x) int-type) rands) int-type))
      (div-prim () (proc-type (map (lambda (x) int-type) rands) int-type))
      ;; Primitivas booleanas: toman N booleanos y retornan un booleano
      (and-prim () (proc-type (map (lambda (x) bool-type) rands) bool-type))
      (or-prim () (proc-type (map (lambda (x) bool-type) rands) bool-type))
      ;; Primitivas relacionales: toman 2 enteros y retornan un booleano
      (less-prim () (proc-type (list int-type int-type) bool-type))
      (lesseq-prim () (proc-type (list int-type int-type) bool-type))
      (more-prim () (proc-type (list int-type int-type) bool-type))
      (moreeq-prim () (proc-type (list int-type int-type) bool-type))
      (eq-prim () (proc-type (list int-type int-type) bool-type))
      (neq-prim () (proc-type (list int-type int-type) bool-type)))))
```

####
**Verificación de tipos en `letrec`**

El `letrec` permite definir procedimientos recursivos. El proceso de verificación requiere:
1. Crear los tipos correspondientes para cada procedimiento definido.
2. Verificar que los cuerpos de los procedimientos cumplan con el tipo de retorno declarado.
3. Retornar el tipo de la expresión principal del `letrec`.

Ejemplo de definición:
`letrec int f(int x, int y) = if >(x,0) then (f -(x,1) +(y,1)) else y`
`bool g(int x) = if >(x,0) (g -(x,1)) else false in (f 20 30)`

```racket
(define type-of-letrec
  (lambda (tprocs procnames ttargs args bodies exp-body tenv)
    (let* ((arg-types (map (lambda (x) (expand-type-expression x)) ttargs))
           (result-types (map (lambda (x) (expand-type-expression x)) tprocs))
           (the-proc-types (map proc-type arg-types result-types))
           (the-env-body (extend-env procnames (list->vector the-proc-types) tenv)))
      (for-each (lambda (ids args-types body result-type)
                  (check-equal-type! 
                   (type-of-expression body (extend-env ids (list->vector args-types) the-env-body))
                   result-type body))
                args arg-types bodies result-types)
      (type-of-expression exp-body the-env-body))))
```

####
**Tratamiento de `set` y `begin`**

La expresión `set` modifica el valor de una variable. Por convención, se asume que retorna `int-type` para evitar la creación de un tipo `void`. Por su parte, `begin` evalúa una secuencia de expresiones y retorna el tipo de la última.

```racket
;; Implementación de set
(define type-of-expression
  (lambda (exp tenv)
    (cases expression exp
      (set-exp (id exp) int-type))))

;; Implementación de begin
(define begin-type-exp
  (lambda (lexp tenv)
    (cond
      [(null? (cdr lexp)) (type-of-expression (car lexp) tenv)]
      [else (begin (type-of-expression (car lexp) tenv)
                   (begin-type-exp (cdr lexp) tenv))])))
```

####
**Intérprete y conceptos teóricos**

El intérprete realiza dos procesos: la **verificación estática** de tipos (antes de la ejecución) y la **evaluación** propiamente dicha.

```racket
(define eval-program
  (lambda (pgm)
    (cases program pgm
      (a-program (exp)
        (if (type? (type-of-expression exp init-tenv))
            (eval-expression exp init-env)
            (eopl:error "The evaluation of types is incorrect!"))))))
```

Conceptos clave:
- **Sistema de tipos**: Garantiza la seguridad de las operaciones.
- **Ambiente de tipos (`tenv`)**: Mantiene el contexto de tipos de las variables durante la verificación.
- **Polimorfismo limitado**: Permite a las primitivas manejar cantidades variables de argumentos siempre que respeten el tipo esperado.
