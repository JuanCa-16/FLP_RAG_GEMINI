####
**Definición de tipos de datos y variantes**

Dos tipos no pueden tener el mismo nombre, ni tampoco dos variantes pueden compartir el mismo nombre, incluso si pertenecen a tipos diferentes. Para cada variante, se crea automáticamente un procedimiento constructor. Si una variante posee `n` campos, su constructor recibe `n` argumentos, verifica que cada uno de ellos satisfaga el predicado asociado y retorna un nuevo valor de dicha variante para el tipo de dato. Asimismo, se vincula un nombre con el formato `nombre-predicado-tipo` a un predicado que determina si un argumento dado pertenece al tipo especificado.

####
**Tipo de dato para expresiones del cálculo λ**

El siguiente código define la estructura para las expresiones del cálculo `lambda` utilizando `define-datatype`:

```racket
(define-datatype lc-exp lc-exp?
  (var-exp (id identifier?))
  (lambda-exp (id identifier?) (body lc-exp?))
  (app-exp (rator lc-exp?) (rand lc-exp?)))
```

####
**Definiciones de tipo s-list y s-exp**

El tipo de dato `s-list` y `s-exp` se definen mediante la siguiente estructura recursiva:

```racket
(define-datatype s-list s-list?
  (empty-s-list)
  (non-empty-s-list (first s-exp?) (rest s-list?)))

(define-datatype s-exp s-exp?
  (symbol-s-exp (sym symbol?))
  (s-list-s-exp (slst s-list?)))
```

####
**Definición alternativa de s-list y list-of**

Una forma alternativa de definir `s-list` junto con un procedimiento auxiliar para validar listas de elementos es:

```racket
(define-datatype s-list s-list?
  (an-s-list (sexps (list-of s-exp?))))

(define (list-of pred)
  (lambda (val)
    (or (null? val)
        (and (pair? val)
             (pred (car val))
             ((list-of pred) (cdr val))))))
```

####
**Uso de la forma cases para tipos de datos**

Para determinar a qué variante de un tipo de dato pertenece un objeto y extraer sus componentes, se utiliza la forma `cases`. Su sintaxis general es:

`(cases nombre-tipo expresion { (nombre-variante ({nombre-campo}*) consecuente) }* (else por-defecto))`

La expresión `cases` evalúa `expresion`, lo que resulta en un valor `v` del tipo `nombre-tipo`. Si `v` es una variante `nombre-variante`, los `nombre-campo` se asocian a los valores correspondientes de `v` y se evalúa el `consecuente`. Si `v` no coincide con las variantes listadas y existe una cláusula `else`, se evalúa la expresión `por-defecto`. Si no existe `else`, el tipo de dato debe estar cubierto exhaustivamente por las variantes definidas.

####
**Ejemplo de uso de cases: occurs-free?**

El siguiente procedimiento utiliza `cases` para determinar si una variable ocurre libre en una expresión del cálculo `lambda`:

```racket
(define occurs-free?
  (lambda (search-var exp)
    (cases lc-exp exp
      (var-exp (var)
        (eqv? var search-var))
      (lambda-exp (bound-var body)
        (and (not (eqv? search-var bound-var))
             (occurs-free? search-var body)))
      (app-exp (rator rand)
        (or (occurs-free? search-var rator)
            (occurs-free? search-var rand))))))
```

####
**Tipo de dato bin-tree y cálculo de suma**

Definición de un árbol binario y un procedimiento para calcular la suma de los valores numéricos almacenados en las hojas:

```racket
(define-datatype bintree bintree?
  (leaf-node (datum number?))
  (interior-node (key symbol?) (left bintree?) (right bintree?)))

(define leaf-sum
  (lambda (tree)
    (cases bintree tree
      (leaf-node (datum) datum)
      (interior-node (key left right)
        (+ (leaf-sum left) (leaf-sum right))))))
```

####
**Estructura de ambientes**

Los ambientes se definen como un tipo de dato que puede estar vacío o extenderse con listas de símbolos y sus valores correspondientes:

```racket
(define-datatype environment environment?
  (empty-env-record)
  (extended-env-record (syms (list-of symbol?))
                       (vals (list-of scheme-value?))
                       (env environment?)))

(define scheme-value? (lambda (v) #t))
```

####
**Búsqueda en ambientes con apply-env**

El procedimiento `apply-env` busca un símbolo en el ambiente, navegando recursivamente por las extensiones hasta encontrar una coincidencia:

```racket
(define apply-env
  (lambda (env sym)
    (cases environment env
      (empty-env-record ()
        (eopl:error 'apply-env "No binding for ~s" sym))
      (extended-env-record (syms vals env)
        (let ((pos (list-find-position sym syms)))
          (if (number? pos)
              (list-ref vals pos)
              (apply-env env sym)))))))
```

####
**Representación y sintaxis: concreta vs. abstracta**

La `sintaxis concreta` es la representación externa de un tipo de dato, conforme a una gramática (como la del cálculo `lambda`). Para procesar estos datos, se requiere convertirlos a una `sintaxis abstracta`, donde los símbolos terminales (como paréntesis) se omiten por no aportar información semántica. Se puede resumir la relación entre ambas mediante:

- `⟨identificador⟩` -> `var-exp (id)`
- `(lambda (⟨identificador⟩) ⟨expresión⟩)` -> `lambda-exp (id body)`
- `(⟨expresión⟩ ⟨expresión⟩)` -> `app-exp (rator rand)`

####
**Árbol de sintaxis abstracta**

La sintaxis abstracta se visualiza comúnmente como un árbol. Para la expresión `(lambda (x) (f (f x)))`, la estructura jerárquica descompone la expresión en su regla gramatical principal (`lambda-exp`) y sus componentes, facilitando que los compiladores o intérpretes procesen el programa recursivamente según sus reglas gramaticales.

####
**Procedimiento unparse-expression**

El procedimiento `unparse-expression` realiza la conversión inversa, transformando un árbol de sintaxis abstracta de nuevo a una representación de lista y símbolos (sintaxis concreta):

```racket
(define unparse-expression
  (lambda (exp)
    (cases expression exp
      (var-exp (id) id)
      (lambda-exp (id body)
        (list 'lambda (list id) (unparse-expression body)))
      (app-exp (rator rand)
        (list (unparse-expression rator) (unparse-expression rand))))))
```

####
**Procedimiento parse-expression**

La tarea de `parsing` consiste en derivar el árbol de sintaxis abstracta a partir de una cadena (o representación concreta). Este procedimiento realiza el análisis sintáctico basándose en la estructura de los datos:

```racket
(define parse-expression
  (lambda (datum)
    (cond
      ((symbol? datum) (var-exp datum))
      ((pair? datum)
       (if (eqv? (car datum) 'lambda)
           (lambda-exp (caadr datum)
                       (parse-expression (caddr datum)))
           (app-exp (parse-expression (car datum))
                    (parse-expression (cadr datum)))))
      (else (eopl:error 'parse-expression "Invalid concrete syntax ~s" datum)))))
```
