El front-end de un compilador se compone de dos procesos principales: Scanning y Parsing. El objetivo del Scanning es analizar el código fuente para crear unidades de caracteres significativos, conocidas como unidades léxicas o tokens, basándose en la especificación léxica de la gramática del lenguaje. Posteriormente, el Parsing toma estas unidades léxicas y las convierte en Árboles de Sintaxis Abstracta (AST). Estos árboles representan la estructura jerárquica del código de acuerdo a la gramática del lenguaje, facilitando su posterior procesamiento y ejecución.
####
El proceso de Scanning, también conocido como análisis léxico, toma el código fuente como una secuencia de caracteres y lo transforma en una lista de unidades léxicas, también llamadas lexemas o tokens. Cada token representa un elemento con significado para el lenguaje de programación. Por ejemplo, en el código `INT C = B + 1;`, el scanner identificaría los siguientes tokens: `INT` (un tipo de dato), `C` (un identificador), `=` (un operador de asignación), `B` (otro identificador), `+` (un operador aritmético) y `1` (un literal numérico). Esta secuencia de tokens es la entrada para la siguiente fase, el Parsing.
####
Durante el proceso de Scanning, ciertos elementos del código fuente son deliberadamente ignorados por no tener relevancia semántica para la ejecución del programa. Estos elementos incluyen los espacios en blanco, los saltos de línea y los comentarios. La función de estos elementos es mejorar la legibilidad del código para los humanos, pero no afectan la estructura lógica del programa. Por lo tanto, el analizador léxico los descarta para simplificar la secuencia de tokens que se envía al parser.
####
La especificación léxica de un lenguaje de programación es el conjunto de reglas formales que define cómo se construyen las unidades significativas o tokens. Estas reglas determinan qué secuencias de caracteres se reconocen como un tipo de dato, un identificador, un operador, una cadena de texto o un número. En esencia, la especificación léxica proporciona al scanner el patrón para identificar y clasificar cada elemento del código fuente en su token correspondiente.
####
En un lenguaje de programación específico, la especificación léxica define reglas concretas. Por ejemplo: 1) Cualquier secuencia de espacios en blanco o saltos de línea se trata como un único separador y se ignora. 2) Los comentarios comienzan con el carácter de porcentaje (%) y se extienden hasta el final de la línea; el scanner los ignora por completo. 3) Un identificador es una secuencia de letras y números que debe comenzar con una letra, utilizado para nombrar variables o procedimientos. Además, existen símbolos específicos para palabras clave y signos de puntuación que también son reconocidos como tokens.
####
Cuando el scanner identifica una secuencia de caracteres como un token, la representa mediante una estructura de datos que contiene, como mínimo, tres informaciones clave: 1) La clase o tipo del token (por ejemplo, `IDENTIFICADOR`, `NÚMERO`, `TIPO_DE_DATO`). 2) El valor o lexema específico del token (el texto original, como el nombre de la variable "C" o el número "1"). 3) La ubicación del token en el código fuente, comúnmente el número de línea, lo cual es fundamental para reportar errores de sintaxis.
####
El proceso de Parsing, o análisis sintáctico, toma como entrada la secuencia de tokens generada por el scanner y la organiza en una estructura jerárquica llamada Árbol de Sintaxis Abstracta (AST). Este árbol representa la estructura gramatical del código. El parser verifica si la secuencia de tokens se ajusta a las reglas definidas en la gramática del lenguaje. Si el código no cumple con la gramática, el parser no puede construir el AST y reporta un error de sintaxis.
####
La gramática de un lenguaje, que define su sintaxis, debe estar completamente especificada con sus símbolos terminales y no terminales para ser coherente. Herramientas como la función `define-datatype` se utilizan para especificar estas gramáticas manualmente. Sin embargo, existen librerías especializadas, como la librería SLG, que automatizan la creación de parsers a partir de una especificación gramatical. Estas herramientas simplifican significativamente el desarrollo de compiladores al integrar la definición de la gramática directamente en el proceso de parsing.
