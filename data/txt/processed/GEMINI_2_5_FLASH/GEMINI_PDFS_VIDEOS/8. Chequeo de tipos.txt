Uno de los conceptos fundamentales de cualquier lenguaje de programación es el concepto de tipo. Los tipos definen las estructuras de almacenamiento disponibles para representar información; además, el tipo determina cómo interpreta el compilador el contenido de la memoria. Un error de tipo se produce cuando se aplica una operación a un dato con características diferentes a las esperadas.

Según la forma en que los lenguajes se comportan frente a los diversos tipos, se clasifican de dos formas:
*   **Lenguajes fuertemente tipados**: requieren definiciones de los tipos y no permiten errores en los tipos. Ejemplo: Pascal.
*   **Lenguajes débilmente tipados**: tienen cierta flexibilidad para errores de tipos (pero sí existen los errores de tipos). Ejemplo: Racket.

####

El análisis de tipos es un paso del procesamiento de un lenguaje de programación. Se realiza para evitar que se presenten errores de tipos en tiempo de ejecución.

Para realizar el análisis de tipos se tienen en cuenta los siguientes aspectos:
1.  Se define un conjunto de tipos para el lenguaje y lo que significa “un valor expresado `v` es de tipo `t`”.
2.  El analizador asigna un tipo para cada expresión en el programa. Si a una expresión `e` se le asigna un tipo `t`, entonces siempre que `e` es interpretada, su valor será de tipo `t`. Esta propiedad hace que un sistema de tipos sea adecuado (*sound*).
3.  El analizador debe inspeccionar también cada invocación de una operación en el programa, con el fin de verificar que los operandos sean del tipo adecuado. Si no se puede determinar si los argumentos son del tipo apropiado, entonces se dice que esta invocación posiblemente genera un error de tipo.
4.  Si se detectan errores de tipo, el analizador puede tomar algunas acciones, las cuales son, normalmente, parte del diseño del lenguaje: puede rehusarse a ejecutar el programa o aplicar medidas correctivas.

####

En el diseño del lenguaje, se definen cuáles tipos habrán, si un valor puede tener más de un tipo, y si los tipos pueden ser determinados en tiempo de ejecución. En algunos lenguajes, en cada ejecución, los valores incluyen una etiqueta para indicar su tipo. Esto es llamado tipado dinámico. Racket es un lenguaje tipado dinámico: las etiquetas son revisadas por procedimientos como `number?`, `string?`, etc.

Las desventajas del tipado dinámico son:
*   Insertar y revisar las etiquetas puede adicionar tiempo en ejecución.
*   No soporta abstracción de datos.

####

En otros lenguajes, los valores pueden representar, por ejemplo, un entero y un carácter. Dichos lenguajes tienen un modelo de ejecución no tipado. Los lenguajes con modelos de ejecución no tipado normalmente no detectan operaciones inapropiadas en tiempo de ejecución. Si las operaciones son aplicadas con datos incorrectos, los resultados no son especificados. En dichos lenguajes se podría hacer la multiplicación de dos caracteres. El resultado de esa operación será lo que el hardware haga con la representación de los caracteres. Esto es llamado un diseño "político".

####

Los lenguajes tipados evitan las dificultades anteriores analizando el programa antes de la ejecución para determinar cualquier llamado particular que pueda resultar en una operación no apropiada en tiempo de ejecución. Esto es llamado chequeo de tipos estático. Si un error es detectado, el analizador podría producir una advertencia, insertar un código de revisión en tiempo de ejecución o rechazar el programa.

####

Los tipos para el lenguaje que se está implementando tienen la siguiente estructura gramatical:
*   `⟨tipo-exp⟩ ::= int-type-exp ()`
*   `⟨tipo-exp⟩ ::= bool-type-exp ()`
*   `⟨tipo-exp⟩ ::= proc-type-exp (arg-texps result-texp)` que se representa como `({⟨tipo-exp⟩}* -> ⟨tipo-exp⟩)`.

Los tipos del lenguaje incluyen tipos base para enteros, booleanos y tipos para procedimientos. El tipo de un procedimiento consiste en los tipos de sus argumentos (separados por `*`) y el tipo de su resultado. Cuando se permite explicitar los tipos en los programas, se suele hacer con expresiones de tipos.

####

La propiedad “un valor expresado `v` es de tipo `t`” se define inductivamente sobre `t` de la siguiente manera:
*   Un valor expresado es de tipo `int` si y solo si es un entero.
*   Un valor expresado es de tipo `bool` si y solo si es un booleano.
*   Un valor expresado es de tipo `(t1 * ... * tn -> t)` si y solo si es un procedimiento (`ProcVal`) que espera exactamente `n` argumentos, y cuando se le aplican `n` argumentos de tipos `t1, ..., tn`, este retorna un valor de tipo `t`.

En este lenguaje, cada valor expresado tiene como máximo un tipo. Sin embargo, no siempre es posible determinar el tipo de un valor en tiempo de ejecución, ya que no siempre es posible determinar el tipo del valor retornado por un procedimiento.

####
<CODIGO_BLOCK>
Se pueden usar las notaciones de tipo definidas para describir valores de Racket. Estos ejemplos describen los tipos de los valores resultantes de las expresiones, no los tipos de las expresiones en sí.

```racket
;; (int -> bool)
;; tipo de even?

;; (int * int -> int)
;; tipo de +

;; (int -> (int -> int))
;; tipo de la expresión:
(lambda (x)
  (lambda (y) (+ x y)))

;; ((int -> int) * int -> bool)
;; tipo de la expresión:
(lambda (f x)
  (even? (f (+ x 1))))
```
</CODIGO_BLOCK>

####

¿Cuál es el tipo de dato de las siguientes expresiones del lenguaje del curso?
1.  `-(32, 26)`
2.  `proc(x) (x 26)`
3.  `proc(x) if x then 1 else 3`
4.  `proc(f, x) if (f x) then 1 else 2`
5.  `proc(x) proc(y) if y then x else 3`

####

El lenguaje que se está diseñando será fuerte y estáticamente tipado, lo que significa que ningún programa que pasa el chequeo de tipos tendrá un error de tipo.

Un error de tipo será:
1.  un intento de aplicar un entero o un booleano a un argumento,
2.  un intento de aplicar un procedimiento o primitiva a un número errado de argumentos,
3.  un intento de aplicar una primitiva que espera un entero a un no-entero, o
4.  un intento de usar un no-booleano como la prueba de una expresión condicional.

Los errores como una división por cero no serán considerados como errores de tipo.

####

Se definirá un procedimiento `type-of-expression` que, dado una expresión `exp` y un ambiente de tipos `tenv` (que asocia cada variable con un tipo), asigna a `exp` un tipo `t` con la propiedad de que: siempre que `exp` es ejecutado en un ambiente en el cual cada variable tiene el tipo especificado por `tenv`, el valor resultante tendrá tipo `t`.

El análisis estará basado en el principio de que si se conocen los tipos de los valores de cada una de las variables en una expresión, se puede deducir el tipo del valor de la expresión.

####

El procedimiento `type-of-expression` se comportará de la siguiente manera:
*   Si la expresión es un número, entonces el resultado será siempre un entero.
*   Si es una expresión booleana (`true` o `false`), el resultado será booleano.
*   Si la expresión es una variable, entonces el resultado es del tipo especificado por `tenv`.

Esto se formaliza como:
*   `(type-of-expression ≪n≫ tenv) = int`
*   `(type-of-expression ≪true≫ tenv) = bool`
*   `(type-of-expression ≪false≫ tenv) = bool`
*   `(type-of-expression ≪id≫ tenv) = (apply-env tenv id)`

####

Una expresión de la forma:
> Si `<condicion1>`
> y `<condicion2>`
> y `<condicion3>`
> ...
> y `<condicionN>`
> entonces `<consecuencia>`

Se conoce como regla de tipado o especificación condicional. En adelante se omitirán los "Si", "y" y "entonces", ya que ellos están implícitos en el formato de la especificación.

####

Si la expresión es una aplicación (de la forma `(rator rand1 ... randn)`), el tipo del resultado se infiere del tipo del operador y los tipos de los operandos. El tipo del operador debe ser obligatoriamente un tipo procedimiento. Si el tipo del operador es `(t1 * t2 * ... * tn -> t)`, entonces debe haber exactamente `n` operandos, y el tipo del i-ésimo operando debe ser `ti`. Si estas condiciones se mantienen, entonces el resultado de la aplicación será el tipo resultado del procedimiento, `t`.

####

La regla de tipado para aplicación de procedimientos o primitivas es:
> `(type-of-expression ≪rator≫ tenv) = (t1 * t2 * ... * tn -> t)`
> `(type-of-expression ≪rand1≫ tenv) = t1`
> `(type-of-expression ≪rand2≫ tenv) = t2`
> ...
> `(type-of-expression ≪randn≫ tenv) = tn`
> ---------------------------------------------------------------------
> `(type-of-expression ≪(rator rand1 rand2 ... randn)≫ tenv) = t`

####

La regla de tipado para las expresiones condicionales es:
> `(type-of-expression ≪test-exp≫ tenv) = bool`
> `(type-of-expression ≪true-exp≫ tenv) = t`
> `(type-of-expression ≪false-exp≫ tenv) = t`
> ---------------------------------------------------------------------
> `(type-of-expression ≪if test-exp then true-exp else false-exp≫ tenv) = t`

####

Para el caso de una expresión que crea un procedimiento (`proc (x1, ..., xn) exp`), se dice que el procedimiento resultante tiene tipo `(t1 * t2 * ... * tn -> t)`, si se cumplen las siguientes condiciones:
*   Cada `xi` está asociado con el tipo `ti`.
*   Tiene exactamente `n` argumentos.
*   Se puede mostrar que cuando el cuerpo `exp` del procedimiento sea ejecutado, producirá un valor de tipo `t`.

####

La regla de tipado para creación de procedimientos es:
> `(type-of-expression ≪exp≫ [x1 = t1, ..., xn = tn]tenv) = t`
> ---------------------------------------------------------------------
> `(type-of-expression ≪proc (x1, ..., xn) exp≫ tenv) = (t1 * t2 * ... * tn -> t)`

Esta regla lleva a un problema: si se trata de encontrar el tipo de una expresión `proc`, ¿cómo se encuentran los tipos `t1, ..., tn` de las variables ligadas? La respuesta es: no hay dónde encontrarlos. Existen dos estrategias para esta situación:
*   **Chequeo de tipos**: se requiere que el programador brinde información sobre los tipos de las variables.
*   **Inferencia de tipos**: el analizador de tipos infiere los tipos de las variables de la forma en que dichas variables son usadas en el programa.

####

En un lenguaje con chequeo de tipos, se requiere que el programador incluya los tipos de todas las variables ligadas. Para hacer chequeo de tipos en el lenguaje del curso, se necesita cambiar la forma en que se escriben los procedimientos (`proc-exp`) y la recursión (`letrec-exp`).

La gramática para expresiones con tipo será ahora:
*   `⟨expresión⟩ ::= proc ({⟨tipo-exp⟩ ⟨identificador⟩}*,) ⟨expresión⟩`
    *   Representado como: `proc-exp (arg-texps ids body)`
*   `⟨expresión⟩ ::= letrec {⟨type-exp⟩ ⟨identificador⟩ ({⟨type-exp⟩ ⟨identificador⟩}*,) = ⟨expresión⟩}* in ⟨expresión⟩`
    *   Representado como: `letrec-exp (result-texps proc-names arg-texpss idss bodies letrec-body)`

####

Se deben añadir las siguientes producciones a la especificación de la gramática del parser:

```racket
(expression
  ("proc" "(" (separated-list type-exp identifier ",") ")" expression)
  proc-exp)

(expression
  ("letrec"
    (arbno
      type-exp
      identifier
      "(" (separated-list type-exp identifier ",") ")"
      "="
      expression)
    "in" expression)
  letrec-exp)
```

####

Así mismo, se añaden las producciones correspondientes a las expresiones `type-exp` a la especificación de la gramática:

```racket
(type-exp
  ("int")
  int-type-exp)

(type-exp
  ("bool")
  bool-type-exp)

(type-exp
  ("(" (separated-list type-exp "*") "->" type-exp ")")
  proc-type-exp)
```

####

De esta manera se pueden escribir programas como el siguiente. Aquí, el tipo resultado de `fact` es `int`, pero el tipo de `fact` como tal es `(int -> int)`.

```
letrec
  int fact (int x) =
    if zero?(x)
    then 1
    else *(x, (fact sub1(x)))
in (fact 5)
```

####

También se pueden escribir programas como el siguiente. Aquí, el tipo resultado de `sum` es `int`, pero el tipo de `sum` como tal es `(bool * int * int -> int)`.

```
let sum = proc(bool b, int x, int y)
  if b then +(x, y) else -(x, y)
in let m = 4
   in let n = 3
      in (sum true m n)
```

####

Ya se ha definido qué es una expresión de tipo. Una expresión de tipo denota un tipo. Pero, ¿qué es un tipo? Ahora se definirán los valores (tipos) que pueden denotar las expresiones de tipo. Para ello se definirá el tipo abstracto de datos "tipo". Un "tipo" es o un tipo atómico con un nombre, o un tipo procedimiento, con una lista de argumentos con tipo y un tipo resultado.

####

El tipo abstracto de datos tendrá la siguiente definición en Racket:

<CODIGO_BLOCK>
```racket
;; Define el tipo de dato algebraico 'type'.
(define-datatype type?
  (atomic-type
    (name symbol?))
  (proc-type
    (arg-types (list-of type?))
    (result-type type?)))

;; Define constantes para los tipos atómicos comunes.
(define int-type (atomic-type 'int))
(define bool-type (atomic-type 'bool))
```
El código define un tipo de dato `type` con dos variantes: `atomic-type` para tipos simples como `int` y `bool`, y `proc-type` para tipos de procedimientos, que contienen una lista de tipos de argumentos y un tipo de resultado. Las constantes `int-type` y `bool-type` son abreviaciones para facilitar su uso.
</CODIGO_BLOCK>

####

Ahora, se debe definir el procedimiento `expand-type-expression` que calcula el tipo denotado por una expresión de tipo. El revisor de tipos (`checker`) llamará a `expand-type-expression` siempre que necesite utilizar el tipo denotado por una expresión de tipo del AST.

####

El procedimiento `expand-type-expression` estará definido así:
<CODIGO_BLOCK>
```racket
(define expand-type-expression
  (lambda (texp)
    (cases type-exp texp
      (int-type-exp () int-type)
      (bool-type-exp () bool-type)
      (proc-type-exp (arg-texps result-texp)
        (proc-type
          (expand-type-expressions arg-texps)
          (expand-type-expression result-texp))))))

(define expand-type-expressions
  (lambda (texps)
    (map expand-type-expression texps)))
```
Este código convierte las representaciones de tipos del árbol sintáctico (`type-exp`) a los valores del tipo de dato `type`. `expand-type-expression` maneja un solo `type-exp`, mientras que `expand-type-expressions` aplica esta conversión a una lista de ellos.
</CODIGO_BLOCK>

####

El revisor de tipos (`checker`) se implementará como un procedimiento que recibe una expresión y un ambiente de tipos, y retorna el tipo correspondiente a ella o un error. El revisor corresponde al procedimiento `type-of-expression` y tendrá cláusulas que implementan las reglas de tipado para las expresiones con tipo.

####

Las primeras cláusulas del procedimiento `type-of-expression` tienen que ver con literales y variables.
*   Si la expresión corresponde a un literal numérico o un booleano, se retorna `int-type` o `bool-type` respectivamente.
*   Si la expresión corresponde a una variable, se retorna el tipo de la variable en el ambiente de tipos en el que se chequea la expresión.
*   Se usa el procedimiento `apply-tenv`, similar a `apply-env` pero con un mensaje de error distinto.

####
<CODIGO_BLOCK>
```racket
(define type-of-expression
  (lambda (exp tenv)
    (cases expression exp
      (lit-exp (number)
        int-type)
      (true-exp ()
        bool-type)
      (false-exp ()
        bool-type)
      (var-exp (id)
        (apply-tenv tenv id))
      ...)))
```
Esta es la implementación inicial de `type-of-expression`, cubriendo los casos base: literales numéricos, booleanos y variables. Para las variables, busca su tipo en el ambiente de tipos (`tenv`).
</CODIGO_BLOCK>

####

La siguiente cláusula del procedimiento `type-of-expression` implementa la regla para expresiones condicionales (`if-exp`), que se formula así:
> `(type-of-expression ≪test-exp≫ tenv) = bool`
> `(type-of-expression ≪true-exp≫ tenv) = t`
> `(type-of-expression ≪false-exp≫ tenv) = t`
> ---------------------------------------------------------------------
> `(type-of-expression ≪if test-exp then true-exp else false-exp≫ tenv) = t`

####
<CODIGO_BLOCK>
```racket
(define type-of-expression
  (lambda (exp tenv)
    (cases expression exp
      ...
      (if-exp (test-exp true-exp false-exp)
        (let ((test-type (type-of-expression test-exp tenv))
              (true-type (type-of-expression true-exp tenv))
              (false-type (type-of-expression false-exp tenv)))
          (check-equal-type! test-type bool-type test-exp)
          (check-equal-type! true-type false-type exp)
          true-type))
      ...)))
```
Esta sección de `type-of-expression` implementa el chequeo para una expresión `if`. Verifica que la condición sea de tipo `bool` y que las ramas de consecuencia y alternativa tengan el mismo tipo. Si todo es correcto, retorna el tipo de las ramas.
</CODIGO_BLOCK>

####

El procedimiento `check-equal-type!` verifica si sus dos primeros argumentos son tipos iguales. El tercer argumento es usado para el reporte de errores.
<CODIGO_BLOCK>
```racket
(define check-equal-type!
  (lambda (t1 t2 exp)
    (if (not (equal? t1 t2))
      (eopl:error 'check-equal-type!
        "Types didn't match: ~s != ~s in~%~s"
        (type-to-external-form t1)
        (type-to-external-form t2)
        exp))))
```
Este procedimiento auxiliar compara dos tipos (`t1`, `t2`) y, si no son iguales, lanza un error que incluye la representación externa de los tipos y la expresión donde ocurrió el fallo.
</CODIGO_BLOCK>

####
<CODIGO_BLOCK>
El procedimiento `check-equal-type!` usa el procedimiento `type-to-external-form` para convertir un tipo en una estructura de lista fácil de leer, por ejemplo `(int * (int -> bool) -> int)`.

```racket
(define type-to-external-form
  (lambda (ty)
    (cases type ty
      (atomic-type (name)
        name)
      (proc-type (arg-types result-type)
        (append
          (list '(')
          (arg-types-to-external-form arg-types)
          '(->)
          (list (type-to-external-form result-type))
          (list ')'))))))
```
Este código convierte la representación interna de un tipo (del `define-datatype`) a un formato S-expression legible para el usuario, útil para mensajes de error.
</CODIGO_BLOCK>

####
<CODIGO_BLOCK>
```racket
(define arg-types-to-external-form
  (lambda (types)
    (cond
      ((null? types) '())
      ((null? (cdr types))
        (list (type-to-external-form (car types))))
      (else
        (cons (type-to-external-form (car types))
          (cons '* (arg-types-to-external-form (cdr types))))))))
```
Procedimiento auxiliar para `type-to-external-form` que formatea una lista de tipos de argumentos, separándolos con el símbolo `*`.
</CODIGO_BLOCK>

####

Retomamos la regla de tipado para creación de procedimientos:
> `(type-of-expression ≪exp≫ [x1 = t1, ..., xn = tn]tenv) = t`
> ---------------------------------------------------------------------
> `(type-of-expression ≪proc (x1, ..., xn) exp≫ tenv) = ((t1 * t2 * ... * tn -> t))`

Esta regla es implementada con ayuda del procedimiento `type-of-proc-exp`.

####

El procedimiento `type-of-proc-exp` recibe una expresión `proc (t1 x1, t2 x2, ..., tn xn) exp`, convierte las expresiones de tipo `t1, ..., tn` en una lista de tipos `arg-types`, revisa el cuerpo `exp` en un ambiente extendido con los tipos de los argumentos, y finalmente construye el tipo de procedimiento resultante (`proc-type`).

####
<CODIGO_BLOCK>
```racket
(define type-of-proc-exp
  (lambda (texps ids body tenv)
    (let ((arg-types (expand-type-expressions texps)))
      (let ((result-type
              (type-of-expression
                body
                (extend-tenv ids arg-types tenv))))
        (proc-type arg-types result-type)))))
```
Esta función determina el tipo de una expresión `proc`. Expande las anotaciones de tipo de los argumentos, y luego calcula el tipo del cuerpo de la función en un nuevo ambiente donde los argumentos están ligados a sus respectivos tipos. El resultado es un `proc-type` que combina los tipos de los argumentos y el tipo del resultado.
</CODIGO_BLOCK>

####
<CODIGO_BLOCK>
```racket
(define type-of-expression
  (lambda (exp tenv)
    (cases expression exp
      ...
      (proc-exp (texps ids body)
        (type-of-proc-exp texps ids body tenv))
      ...)))
```
Esta cláusula en `type-of-expression` delega el chequeo de tipo de las expresiones de procedimiento (`proc-exp`) al procedimiento auxiliar `type-of-proc-exp`.
</CODIGO_BLOCK>

####

Para aplicaciones de primitivas o de procedimientos, teníamos la siguiente regla de tipado:
> `(type-of-expression ≪rator≫ tenv) = (t1 * t2 * ... * tn -> t)`
> `(type-of-expression ≪rand1≫ tenv) = t1`
> ...
> `(type-of-expression ≪randn≫ tenv) = tn`
> ---------------------------------------------------------------------
> `(type-of-expression ≪(rator rand1 ... randn)≫ tenv) = t`

####

`type-of-expression` encuentra el tipo del operador y luego llama a un procedimiento auxiliar `type-of-application` para aplicar la regla de tipado. El procedimiento `type-of-application` primero revisa si el tipo del operador es un tipo procedimiento. Luego, revisa si el número de argumentos esperados por el procedimiento es igual al número dado y después, en un ciclo `for-each`, revisa si el tipo de cada argumento esperado es igual al tipo del operando correspondiente. Si la revisión tiene éxito, entonces el tipo de la aplicación es el tipo resultado del procedimiento.

####
<CODIGO_BLOCK>
```racket
(define type-of-application
  (lambda (rator-type rand-types rator-exp rands-exp exp)
    (cases type rator-type
      (proc-type (arg-types result-type)
        (if (= (length arg-types) (length rand-types))
            (begin
              (for-each
                check-equal-type!
                rand-types
                arg-types
                rands-exp)
              result-type)
            (eopl:error 'type-of-expression
              "Wrong number of arguments in expression ~s:~%expected ~s~%got ~s"
              exp
              (map type-to-external-form arg-types)
              (map type-to-external-form rand-types))))
      (else
        (eopl:error 'type-of-expression
          "Rator not a proc type in expression:~%~s~%had rator type ~s"
          rator-exp
          (type-to-external-form rator-type))))))
```
Este procedimiento verifica el tipado de una aplicación. Asegura que el operador (`rator`) sea un procedimiento, que el número de argumentos sea correcto, y que el tipo de cada argumento provisto coincida con el tipo esperado. Si todas las verificaciones pasan, devuelve el tipo de resultado del procedimiento.
</CODIGO_BLOCK>

####

Para determinar el tipo del operador cuando se trata de una aplicación de primitivas, se utiliza el procedimiento auxiliar `type-of-primitive`. Este procedimiento recibe un valor del tipo de dato `primitive` y crea un `proc-type` de acuerdo a ese valor.

####
<CODIGO_BLOCK>
```racket
(define type-of-primitive
  (lambda (prim)
    (cases primitive prim
      (add-prim ()
        (proc-type (list int-type int-type) int-type))
      (incr-prim ()
        (proc-type (list int-type) int-type))
      (zero-test-prim ()
        (proc-type (list int-type) bool-type))
      ...)))
```
Esta función asigna un tipo de procedimiento a cada operación primitiva del lenguaje. Por ejemplo, a `add-prim` le asigna el tipo `(int * int -> int)`.
</CODIGO_BLOCK>

####
<CODIGO_BLOCK>
Ahora, se añaden las cláusulas correspondientes a la aplicación de primitivas y procedimientos al `type-of-expression`:

```racket
(define type-of-expression
  (lambda (exp tenv)
    (cases expression exp
      ...
      (primapp-exp (prim rands)
        (type-of-application
          (type-of-primitive prim)
          (types-of-expressions rands tenv)
          prim
          rands
          exp))
      (app-exp (rator rands)
        (type-of-application
          (type-of-expression rator tenv)
          (types-of-expressions rands tenv)
          rator
          rands
          exp))
      ...)))
```
Estas cláusulas manejan la aplicación de funciones. Para primitivas (`primapp-exp`), obtiene el tipo de la primitiva y lo verifica. Para procedimientos definidos por el usuario (`app-exp`), primero determina el tipo del operador (`rator`) y luego verifica la aplicación.
</CODIGO_BLOCK>

####
<CODIGO_BLOCK>
El procedimiento `types-of-expressions` calcula los tipos de una lista de expresiones.

```racket
(define types-of-expressions
  (lambda (rands tenv)
    (map
      (lambda (exp)
        (type-of-expression exp tenv))
      rands)))
```
</CODIGO_BLOCK>

####

La regla de tipado para expresiones `let` es:
> `(type-of-expression ≪e1≫ tenv) = t1`
> ...
> `(type-of-expression ≪en≫ tenv) = tn`
> `(type-of-expression ≪body≫ [x1 = t1, ..., xn = tn]tenv) = t`
> ---------------------------------------------------------------------
> `(type-of-expression ≪let x1 = e1 ... xn = en in body≫ tenv) = t`

Esta regla es implementada con ayuda del procedimiento `type-of-let-exp`.

####

El procedimiento `type-of-let-exp` encuentra el tipo de las expresiones en las declaraciones del `let` y extiende el ambiente de tipos con las variables del `let` y los tipos encontrados. Luego, `type-of-let-exp` determina el tipo de la expresión correspondiente a su cuerpo en el ambiente de tipos extendido.

####
<CODIGO_BLOCK>
```racket
(define type-of-let-exp
  (lambda (ids rands body tenv)
    (let ((tenv-for-body
            (extend-tenv
              ids
              (types-of-expressions rands tenv)
              tenv)))
      (type-of-expression body tenv-for-body))))
```
Esta función determina el tipo de una expresión `let`. Calcula los tipos de las expresiones de la derecha de las asignaciones y luego evalúa el tipo del cuerpo (`body`) en un ambiente extendido donde las nuevas variables están ligadas a esos tipos.
</CODIGO_BLOCK>

####
<CODIGO_BLOCK>
Se añade la cláusula respectiva a la definición del procedimiento `type-of-expression`:

```racket
(define type-of-expression
  (lambda (exp tenv)
    (cases expression exp
      ...
      (let-exp (ids rands body)
        (type-of-let-exp ids rands body tenv))
      ...)))
```
</CODIGO_BLOCK>

####

Una expresión `letrec` se ve de esta manera:
```
letrec
  t1 p1(t11 x11, ..., t1n1 x1n1) = e1
  t2 p2(t21 x21, ..., t2n2 x2n2) = e2
  ...
in body
```
En la cual se declaran un conjunto de procedimientos `p1, p2, ...` con cuerpos `e1, e2, ...`. El tipo de `pi` será `(ti1 * ti2 * ... * tin -> ti)`.

####

El cuerpo del `letrec` y cada uno de los cuerpos de los procedimientos deben ser revisados en un ambiente donde cada variable tenga el tipo correcto. Como el procedimiento `pi` tiene tipo `(ti1 * ti2 * ... * tin -> ti)`, el cuerpo del `letrec` debe ser revisado en el ambiente:
`tenv-body = [p1 = (t11 * t12 * ... -> t1), p2 = (t21 * t22 * ... -> t2), ...]tenv`

Cada cuerpo de los procedimientos, `ei`, se debe revisar en el ambiente:
`tenv-i = [xi1 = ti1, xi2 = ti2, ...]tenv-body`

####

Luego, la regla de tipado para expresiones `letrec` es:
> `(type-of-expression ≪e1≫ tenv1) = t1`
> `(type-of-expression ≪e2≫ tenv2) = t2`
> ...
> `(type-of-expression ≪en≫ tenvn) = tn`
> `(type-of-expression ≪body≫ tenv-body) = t`
> ---------------------------------------------------------------------
> `(type-of-expression ≪letrec ... in body≫ tenv) = t`

####

La cláusula correspondiente a una expresión `letrec` en `type-of-expression` es:

```racket
(letrec-exp (result-texps proc-names texpss idss bodies letrec-body)
  (type-of-letrec-exp
    result-texps proc-names
    texpss idss bodies
    letrec-body tenv))
```

Donde se utiliza el procedimiento auxiliar `type-of-letrec-exp`.

####

`type-of-letrec-exp` es un procedimiento que, inicialmente, convierte las expresiones de tipo de los argumentos y el resultado a tipos. Luego liga la lista de los tipos de los procedimientos a la variable `the-proc-types` y el ambiente del cuerpo de `letrec` a `tenv-for-body`. Finalmente, computa el tipo del cuerpo de cada procedimiento y lo compara con el tipo del resultado especificado.

####
<CODIGO_BLOCK>
```racket
(define type-of-letrec-exp
  (lambda (result-texps proc-names texpss idss bodies letrec-body tenv)
    (let ((arg-typess (map expand-type-expressions texpss))
          (result-types (expand-type-expressions result-texps)))
      (let ((the-proc-types (map proc-type arg-typess result-types)))
        (let ((tenv-for-body
                (extend-tenv proc-names the-proc-types tenv)))
          (for-each
            (lambda (ids arg-types body result-type)
              (check-equal-type!
                (type-of-expression
                  body
                  (extend-tenv ids arg-types tenv-for-body))
                result-type
                body))
            idss arg-typess bodies result-types)
          (type-of-expression letrec-body tenv-for-body))))))
```
Esta función gestiona el chequeo de tipos para `letrec`. Primero, construye el ambiente `tenv-for-body` que contiene los tipos de todos los procedimientos declarados, permitiendo la recursión mutua. Luego, itera sobre cada procedimiento, verificando que el tipo de su cuerpo coincida con el tipo de resultado declarado. Finalmente, determina el tipo del cuerpo principal del `letrec` en el ambiente extendido.
</CODIGO_BLOCK>
