Cada vez que se define un conjunto de valores, se está definiendo un nuevo tipo de dato. En un tipo de dato se definen los valores, representaciones y operaciones sobre el mismo. La representación de los datos es bastante compleja. Para cambiar la representación de los datos se lleva a cabo una tarea llamada Abstracción de Datos.

####

La abstracción de datos divide en dos los tipos de datos: interfaz e implementación. La interfaz de un tipo de datos dice qué representa el tipo de dato, sus operaciones y las propiedades de dichas operaciones. La implementación proporciona una representación específica y un código para las operaciones que hacen uso de esa representación.

####

Un tipo de dato que es abstracto se denomina Tipo Abstracto de Dato (TAD). El resto del programa fuera del tipo de dato, llamado el cliente del tipo de dato, manipula el nuevo dato solo mediante las operaciones especificadas en la interfaz. El código del cliente es independiente de la representación si manipula los valores del tipo de dato solo a través de los procedimientos en su interfaz.

####

Todo lo relacionado sobre el dato representado debe estar en el código de la implementación. La parte más importante de la implementación es la especificación de cómo los datos son representados. Se utiliza la notación ⌈v⌉ para la representación del dato v.

####

Para la representación de números naturales, una especificación no indica cómo se representan, sino qué operaciones definen el tipo.
<IMG_BLOCK>
Especificación matemática de las operaciones para un tipo de dato de números naturales: `(zero)` se representa como `[0]`, `(is-zero? [n])` es verdadero si `n=0`, `(successor [n])` es `[n+1]`, y `(predecessor [n+1])` es `[n]` para `n>0`.
</IMG_BLOCK>
A partir de esta especificación se pueden escribir programas para la manipulación de los datos, sin importar su representación.

####

Un procedimiento como `plus` puede ser definido usando solo la interfaz de los números naturales. La siguiente implementación recursiva satisface la propiedad `(plus ⌈x⌉ ⌈y⌉) = ⌈x + y⌉`.
<CODIGO_BLOCK>
;; Define la suma de dos números naturales representados abstractamente.
(define (plus x y)
  (if (is-zero? x)
      y
      (successor (plus (predecessor x) y))))
</CODIGO_BLOCK>

####

Existen tres posibles representaciones para los enteros no negativos. La primera es la representación unaria, donde un entero no negativo `n` es representado por una lista de `n` símbolos `#t`.
`⌈0⌉ = '()`
`⌈n + 1⌉ = (cons #t ⌈n⌉)`
En esta representación, se satisface la especificación escribiendo:
<CODIGO_BLOCK>
;; Implementación de la interfaz de números naturales
;; usando una representación unaria (listas).
(define zero '())
(define is-zero? null?)
(define successor
  (lambda (n)
    (cons #t n)))
(define predecessor cdr)
</CODIGO_BLOCK>

####

Una segunda alternativa es la representación de números de Scheme, donde se usa la representación interna de números del lenguaje.
`⌈n⌉ = n`
Se definen las entidades como:
<CODIGO_BLOCK>
;; Implementación de la interfaz de números naturales
;; usando los números nativos de Racket.
(define zero 0)
(define is-zero? zero?)
(define successor
  (lambda (n)
    (+ n 1)))
(define predecessor
  (lambda (n)
    (- n 1)))
</CODIGO_BLOCK>

####

Una tercera opción es la representación Bignum. Los números son representados en base `N`, para algún entero grande `N`. Dicha representación es una lista que consiste de números entre 0 y `N - 1`, donde el primer elemento es el dígito menos significativo.
`⌈n⌉ = '()` si `n = 0`
`⌈n⌉ = (cons r ⌈q⌉)` si `n = qN + r`, `0 ≤ r < N`
Luego si `N = 16`, entonces:
`⌈33⌉ = (1 2)` porque `33 = 1 * 16^0 + 2 * 16^1`
`⌈258⌉ = (2 0 1)` porque `258 = 2 * 16^0 + 0 * 16^1 + 1 * 16^2`

####

Cuando es usada la abstracción de datos, los programas tienen la propiedad de independencia de representación. Se presentan algunas estrategias para representar tipos de datos ilustradas usando el tipo de dato "ambiente". Un ambiente asocia un valor con cada elemento de un conjunto finito de variables. Un ambiente puede ser usado para asociar las variables con sus valores en la implementación de un lenguaje de programación.

####

Las variables pueden ser representadas de cualquier manera, siempre y cuando sea posible chequear la igualdad entre dos variables. Las variables se pueden representar mediante símbolos, cadenas, referencias en una tabla hash o incluso mediante números.

####

Un ambiente es una función cuyo dominio es un conjunto finito de variables y cuyo rango es el conjunto de todos los los valores de Scheme. De acuerdo a la notación matemática, los ambientes representan todos los conjuntos de la forma `{(s₁, v₁), ..., (sₙ, vₙ)}`, donde los `sᵢ` son símbolos diferentes y los `vᵢ` son valores de Scheme.

####

La interfaz del tipo de dato ambiente tiene tres procedimientos:
`(empty-env)` = ⌈∅⌉
`(apply-env ⌈f⌉ s)` = `f(s)`
`(extend-env var v ⌈f⌉)` = ⌈g⌉, donde `g(s') = v` si `s' = var`, y `g(s') = f(s')` de otra forma.

####

El procedimiento `empty-env` debe producir una representación del ambiente vacío. El procedimiento `apply-env` aplica una representación de un ambiente a un argumento (una variable) para obtener su valor. El procedimiento `(extend-env var val env)` produce un nuevo ambiente que se comporta como `env`, excepto que su valor en el símbolo `var` es `val`.
<CODIGO_BLOCK>
;; Ejemplo de construcción de un ambiente anidado.
;; La variable 'y' es sombreada.
(extend-env 'd 6
  (extend-env 'y 8
    (extend-env 'x 7
      (extend-env 'y 14
        (empty-env)))))
</CODIGO_BLOCK>

####

Una forma de implementar ambientes es mediante una representación como estructura de datos. Cada ambiente puede ser construido mediante una expresión en la siguiente gramática:
`<env-exp> ::= (empty-env)`
`<env-exp> ::= (extend-env <identificador> <scheme-value> <env-exp>)`
De acuerdo a esta gramática, los ambientes pueden ser representados como listas en Scheme.

####

La representación de ambientes como estructura de datos se implementa con los siguientes constructores.
`empty-env : () → Env`
<CODIGO_BLOCK>
;; Constructor para un ambiente vacío.
(define (empty-env)
  (list 'empty-env))
</CODIGO_BLOCK>
`extend-env : Var x SchemeValue x Env → Env`
<CODIGO_BLOCK>
;; Constructor para extender un ambiente con una nueva variable y su valor.
(define (extend-env var val env)
  (list 'extend-env var val env))
</CODIGO_BLOCK>

####

Para la representación como estructura de datos, el observador `apply-env` busca recursivamente la variable en la estructura de lista.
`apply-env : Env x Var → SchemeValue`
<CODIGO_BLOCK>
;; Observador que busca el valor de una variable en un ambiente.
(define (apply-env env search-var)
  (cond
    ((eqv? (car env) 'empty-env)
     (report-no-binding-found search-var))
    ((eqv? (car env) 'extend-env)
     (let ((saved-var (cadr env))
           (saved-val (caddr env))
           (saved-env (cadddr env)))
       (if (eqv? search-var saved-var)
           saved-val
           (apply-env saved-env search-var))))
    (else
     (report-invalid-env env))))
</CODIGO_BLOCK>

####

Una alternativa es la representación procedimental. La interfaz del tipo de dato ambiente tiene exactamente una entidad observadora: `apply-env`. Esto permite representar un ambiente como un procedimiento que toma una variable y retorna su valor asociado. Se define `empty-env` y `extend-env` de tal manera que retornan un procedimiento que, al ser aplicado, devuelve el valor asociado a la variable en el ambiente.

####

En la representación procedimental, los constructores `empty-env` y `extend-env` devuelven procedimientos (lambdas).
`empty-env : () → Env`
<CODIGO_BLOCK>
;; Constructor que devuelve un procedimiento representando un ambiente vacío.
(define (empty-env)
  (lambda (search-var)
    (report-no-binding-found search-var)))
</CODIGO_BLOCK>
`extend-env : Var x SchemeValue x Env → Env`
<CODIGO_BLOCK>
;; Constructor que devuelve un procedimiento que representa un ambiente extendido.
(define (extend-env saved-var saved-val saved-env)
  (lambda (search-var)
    (if (eqv? search-var saved-var)
        saved-val
        (apply-env saved-env search-var))))
</CODIGO_BLOCK>

####

En la representación procedimental, el observador `apply-env` es muy simple, ya que el ambiente en sí mismo es el procedimiento de búsqueda.
`apply-env : Env x Var → SchemeValue`
<CODIGO_BLOCK>
;; Observador que aplica el ambiente (un procedimiento) a la variable.
(define (apply-env env search-var)
  (env search-var))
</CODIGO_BLOCK>

####

Hasta el momento hemos visto cómo definir tipos de datos. Ahora consideraremos tipos de datos recursivos, como las expresiones del cálculo lambda.
<CODIGO_BLOCK>
;; Gramática formal para las expresiones del cálculo lambda (Lc-exp).
;; <Lc-exp> ::= <var-exp>
;; <var-exp> ::= <identifier>
;; <Lc-exp> ::= <lambda-exp>
;; <lambda-exp> ::= (lambda (<identifier>) <Lc-exp>)
;; <Lc-exp> ::= <app-exp>
;; <app-exp> ::= (<Lc-exp> <Lc-exp>)
</CODIGO_BLOCK>

####

Para una representación basada en listas de las expresiones del cálculo lambda, se pueden usar las siguientes estructuras.
<CODIGO_BLOCK>
;; Representación en lista de: (lambda (x) x)
(list 'lambda 'x 'x)

;; Representación en lista de: ((lambda (x) (lambda (y) (x y))) x)
(list
 (list 'lambda 'x (list 'lambda 'y (list 'x 'y)))
 'x)
</CODIGO_BLOCK>

####

Así mismo, hemos intentado manipular y operar tipos de datos recursivos. Por ejemplo, para las expresiones del cálculo λ se ha definido el procedimiento `occurs-free?`, que determina si una variable aparece libre en una expresión.
<CODIGO_BLOCK>
;; Procedimiento que verifica si una variable 'var' aparece libre en 'exp',
;; asumiendo una representación de 'exp' como lista.
(define (occurs-free? var exp)
  (cond
    ;; Caso var-exp
    [(symbol? exp)
     (eqv? exp var)]
    ;; Caso lambda-exp
    [(eqv? (car exp) 'lambda)
     (and
      (not (eqv? (cadr exp) var))
      (occurs-free? var (caddr exp)))]
    ;; Caso app-exp
    [else
     (or
      (occurs-free? var (car exp))
      (occurs-free? var (cadr exp)))]))
</CODIGO_BLOCK>

####

No obstante, esta definición de `occurs-free?` es difícil de leer (es difícil decir que `(cadr exp)` se refiere a la declaración de una variable en una expresión lambda o que `(caddr exp)` se refiere a su cuerpo). Esta definición establece una fuerte dependencia con la implementación de las expresiones del cálculo λ como listas.

####

Se puede mejorar estos aspectos introduciendo interfaces para la creación y manipulación de datos de un cierto tipo. Una interfaz para un tipo de dato consta de procedimientos constructores y procedimientos observadores. Los procedimientos observadores pueden ser predicados o extractores.

####

Para las expresiones del cálculo λ tenemos la siguiente interfaz:
**Constructores:**
`var-exp : Var → Lc-exp`
`lambda-exp : Var x Lc-exp → Lc-exp`
`app-exp : Lc-exp x Lc-exp → Lc-exp`

**Predicados:**
`var-exp? : Lc-exp → Bool`
`lambda-exp? : Lc-exp → Bool`
`app-exp? : Lc-exp → Bool`

####

**Extractores de la interfaz para expresiones del cálculo λ:**
`var-exp->var : Lc-exp → Var`
`lambda-exp->bound-var : Lc-exp → Var`
`lambda-exp->body : Lc-exp → Lc-exp`
`app-exp->rator : Lc-exp → Lc-exp`
`app-exp->rand : Lc-exp → Lc-exp`

####

Ahora es posible escribir una versión de `occurs-free?` que depende solo de la interfaz, haciéndola independiente de la representación subyacente.
<CODIGO_BLOCK>
;; Versión de 'occurs-free?' que utiliza la interfaz del tipo de dato
;; Lc-exp, haciéndola independiente de la representación.
(define (occurs-free? search-var exp)
  (cond
    ((var-exp? exp)
     (eqv? search-var (var-exp->var exp)))
    ((lambda-exp? exp)
     (and
      (not (eqv? search-var (lambda-exp->bound-var exp)))
      (occurs-free? search-var (lambda-exp->body exp))))
    ((app-exp? exp)
     (or
      (occurs-free? search-var (app-exp->rator exp))
      (occurs-free? search-var (app-exp->rand exp))))))
</CODIGO_BLOCK>

####

La interfaz para el tipo de dato ambiente consta de los constructores `empty-env` y `extend-env` y del procedimiento observador `apply-env`. La interfaz para el tipo de dato lista consta de los constructores `empty-list` y `cons`, de los procedimientos extractores `car` y `cdr` y del predicado `list?`.

####

Receta general para el diseño de interfaces de datos recursivos:
1.  Incluir un constructor para cada clase de dato (regla de producción) en el tipo de dato.
2.  Incluir un predicado para cada clase de dato en el tipo de dato.
3.  Incluir un extractor para cada pieza de dato pasada a un constructor del tipo de dato.
