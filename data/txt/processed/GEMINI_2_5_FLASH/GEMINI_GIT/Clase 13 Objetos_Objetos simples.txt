Los objetos simples son una implementación basada en listas de partes, cada parte representa una clase en una cadena herencia. Por ejemplo:

```
class c1 extends object
    field x
    field y
    method initialize () 1

class c2 extends c1
    field z
    field w
    method initialize () 1

let o1 = new c1 ()
    o2 = new c2 ()
in
??
```

###
Cuando es `o1` se muestra su representación, donde una parte contiene el nombre de la clase y los valores de los campos:

```
(#(struct:a-part c1 #(0 0)))
```

###
Cuando se crea `o2` (instancia de `c2` que extiende `c1`), la representación muestra las partes de la cadena de herencia. El primer elemento es la clase `c2` y el segundo la clase `c1`, asumiendo `object` como la lista vacía:

```
(#(struct:a-part c2 #(0 0)) #(struct:a-part c1 #(0 0)))
```

###
Consideremos una cadena de herencia más compleja con `c3` extendiendo `c2`:

```
class c1 extends object
    field x
    field y
    method initialize () 1

class c2 extends c1
    field z
    field w
    method initialize () 1

class c3 extends c2
    field a
    field b
    method initialize () 1

let o3 = new c3 ()
in
o3
```

En este caso, la representación de `o3` muestra la siguiente estructura:

```
(#(struct:a-part c3 #(0 0)) #(struct:a-part c2 #(0 0)) #(struct:a-part c1 #(0 0)))
```

###
Entonces, la construcción de objetos con herencia se puede visualizar como una secuencia de partes, donde la clase de menor jerarquía es el primer elemento y `object` se representa como una lista vacía.

```mermaid
flowchart LR
    A["C3
    a = 0, b = 0
    "]
    B["C2
    z = 0, w = 0
    "]
    C["C1
    x = 0, y = 0
    "]
    D["object
    '()
    "]
    A --> B
    B --> C
    C --> D
```

Esta implementación es utilizada por C++ para representar objetos.

###
A continuación, se definen clases más elaboradas (`p1`, `p2`, `p3`) con campos y métodos que incluyen llamadas a `super` y modificaciones de estado:

```
class p1 extends object
    field a
    field b
    method initialize (x,y)
        begin
            set a = x;
            set b = y
        end
    method sum (x)
        begin
            set a = +(a,1);
            +(a,b,x, send self m1())
        end
    method m1 () 1

class p2 extends p1
    field c
    field d
    field e
    method initialize (x,y,z)
        begin
            super initialize (x,y);
            set c = +(x,1);
            set d = +(y,2);
            set e = z
        end
    method sum (s)
        +(c,d,e, send self m1())

class p3 extends p2
    field f
    field g
    method initialize (u)
        begin
            set f = u;
            set g = +(u,1);
            super initialize (u, +(u,1), +(u,2))
        end
    method sum (a)
        +(a, super sum (+(a,2)))
    method m1 () 10

let o1 = new p1 (2,3)
    o2 = new p2 (3,4,5)
    o3 = new p3 (6)
in
list (send o1 sum (2), send o2 sum (3), send o3 sum (4))
```

Al ejecutar el código anterior, obtenemos los siguientes resultados:

```
(9 16 39)
```

###
Vamos a dibujar los ambientes de los llamados de los objetos, comenzando con la creación de `o1 = new p1(2,3)`.
El diagrama de herencia y su representación son:

```mermaid
flowchart LR
    A["c1
    a = 2, b = 3
    "]
    B["object"]
    A --> B
```

```
(#(struct:a-part p1 #(2 3)))
```

###
Continuamos con la creación de `o2 = new p2(3,4,5)`.
Su diagrama de herencia y representación son:

```mermaid
flowchart LR
    A["c2
    c = 4 d = 6 e = 5
    "]
    B["c1
    a = 3, b = 4
    "]
    C["object"]
    B --> C
    A --> B
```

```
(#(struct:a-part p2 #(4 6 5)) #(struct:a-part p1 #(3 4)))
```

###
Finalmente, para `o3 = new p3(6)`, el diagrama de herencia y representación son:

```mermaid
flowchart LR
    A["c3
    f = 6, g = 7
    "]
    B["c2
    c = 7 d = 9 e = 8
    "]
    C["c1
    a = 6, b = 7
    "]
    D["object"]
    B --> C
    A --> B
    C --> D
```

```
(#(struct:a-part p3 #(6 7)) #(struct:a-part p2 #(7 9 8)) #(struct:a-part p1 #(6 7)))
```

###
**Ejecución de métodos**

Cuando se invoca un método, se tiene un diagrama de clases independiente (propiedad en encapsulación). Cada llamado es independiente y no se afectan entre sí, a excepción de si uno de ellos ha modificado los campos del objeto.

Comencemos con el llamado `send o1 sum(2)`. El diagrama de ambientes para esta invocación es:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1
    a,b
    2,3
    "]
    C["envsum
    x,super,self
    2,object,o1
    "]
    A --> B
    B --> C
```

Sobre el ambiente `envsum` se ejecuta el siguiente código:

```
begin
    set a = +(a,1);
    +(a,b,x, send self m1())
end
```

###
Al hacer `set a = +(a,1)`, se modifica la representación interna del objeto `o1`. Ahora `a` vale `3`:

```mermaid
flowchart LR
    A["c1
    a = 3, b = 3
    "]
    B["object"]
    A --> B
```

Esto cambia el ambiente `env1`:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1
    a,b
    3,3
    "]
    C["envsum
    x,super,self
    2,object,o1
    "]
    A --> B
    B --> C
```

###
Luego, cuando ejecutamos `+(a,b,x, send self m1())`, se convierte en `+(3,3,2, send o1 m1())`.
La llamada a `send o1 m1()` utiliza el siguiente ambiente:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1
    a,b
    3,3
    "]
    C["envm1
    super,self
    object,o1
    "]
    A --> B
    B --> C
```

El método `m1` de `p1` devuelve `1`. Por lo tanto, la expresión final es `+(3,3,2,1) = 9`.

###
Ahora, consideremos el llamado `send o2 sum(3)`. El diagrama de ambientes para este caso es:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1
    a,b
    3,4
    "]
    C["env2
    c,d,e
    4,6,5
    "]
    D["envsum
    s,super,self
    3,p1,o2
    "]
    A --> B
    B --> C
    C --> D
```

Sobre el ambiente `envsum` se ejecuta la expresión:

```
+(c,d,e, send self m1())
```

Que se traduce a `+(4,6,5, send o2 m1())`.

###
La llamada a `send o2 m1()` utiliza el siguiente ambiente. El método `m1` para `p2` hereda de `p1` y devuelve `1`.

```mermaid
flowchart LR
    A["empty-env"]
    B["env1
    a,b
    3,4
    "]
    C["envm1
    super,self
    p1,object, o2
    "]
    A --> B
    B --> C
```

Aquí se evalúa la expresión `1` (del método `m1` de `p1`). Por lo tanto, la expresión final es `+(4,6,5, 1) = 16`.

###
Finalmente, analizamos `send o3 sum(4)`. El diagrama de ambientes para la llamada al método `sum` de `o3` es:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1
    a,b
    6,7
    "]
    C["env2
    c,d,e
    7,9,8
    "]
    D["env3
    f,g
    6,7
    "]
    E["envsum
    a,super,self
    4,p2,o3
    "]
    A --> B
    B --> C
    C --> D
    D --> E
```

Vamos a ejecutar el cuerpo del método:

```
+(a, super sum (+(a,2)))
```

Esto se convierte en `+(4, super sum (6))`.

###
Ahora se invoca `super sum(6)`. Este método se encuentra en `p2` (la superclase de `p3`). El diagrama de ambientes para esta llamada es:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1
    a,b
    6,7
    "]
    C["env2
    c,d,e
    7,9,8
    "]
    D["envsum
    s,super,self
    6,p1,o3
    "]
    A --> B
    B --> C
    C --> D
```

Sobre el ambiente `envsum` de `p2` se ejecuta:

```
+(c,d,e, send self m1())
```

Que se traduce a `+(7,9,8, send o3 m1())`.

###
Luego, se invoca `send o3 m1()`. En este caso, `o3` es una instancia de `p3`, y `p3` tiene su propio método `m1` que devuelve `10`. El ambiente para esta llamada es:

```mermaid
flowchart LR
    A["empty-env"]
    B["env1
    a,b
    6,7
    "]
    C["env2
    c,d,e
    7,9,8
    "]
    D["env3
    f,g
    6,7
    "]
    E["envm1
    super,self
    p2,o3
    "]
    A --> B
    B --> C
    C --> D
    D --> E
```

Bajo el ambiente `envm1` se ejecuta `10`.

Por lo que, al evaluar las expresiones anidadas:
`+(7,9,8, 10)` resulta en `34`.
Finalmente, `+(4, 34)` resulta en `38`.
