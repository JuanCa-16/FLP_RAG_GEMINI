El procedimiento `expand-type-of-expression` tiene como objetivo calcular el tipo denotado por una expresión de tipo proveniente de la especificación gramatical. Por ejemplo, si recibe una expresión de tipo numérico, la convierte en una representación interna como un `atomic-type` con la etiqueta `INT`. El revisor de tipos (`type-checker`) invocará esta función siempre que necesite la representación interna de un tipo definido en la gramática.

La implementación se realiza mediante un `case` sobre la expresión de tipo:
-   Para `int-type-expression`, retorna `int-type`.
-   Para `bool-type-expression`, retorna `bool-type`.
-   Para `proc-type-expression`, que contiene los tipos de los argumentos y el tipo del resultado, construye un `proc-type`. Para ello, aplica recursivamente `expand-type-of-expression` a cada tipo de argumento (usando una función auxiliar `expand-type-expressions` que mapea la función sobre una lista) y al tipo de resultado.

```scheme
(define expand-type-of-expression
  (lambda (type-exp)
    (cases type-expression type-exp
      (int-type-expression () (int-type))
      (bool-type-expression () (bool-type))
      (proc-type-expression (arg-tes result-te)
        (proc-type
          (expand-type-expressions arg-tes)
          (expand-type-of-expression result-te))))))

(define expand-type-expressions
  (lambda (arg-tes)
    (map
      (lambda (x) (expand-type-of-expression x))
      arg-tes)))
```

####
El revisor de tipos se implementa como un procedimiento principal llamado `type-of-expression`, que recibe una expresión del lenguaje y un ambiente de tipos (`tenv`). Su función es retornar el tipo correspondiente a dicha expresión o señalar un error de tipos si la expresión no es válida según las reglas. Este procedimiento tendrá una cláusula para cada regla de producción de la gramática de expresiones, asociando a cada estructura sintáctica su tipo correspondiente.

####
Las primeras cláusulas de `type-of-expression` gestionan los casos base: literales y variables.
-   Para una `literal-expression` que contiene un número, el tipo retornado es `int-type`.
-   Para una `literal-expression` que contiene un valor booleano (`#t` o `#f`), el tipo retornado es `bool-type`.
-   Para una `var-expression`, que representa una variable, se busca su tipo en el ambiente de tipos (`tenv`) utilizando el procedimiento `apply-tenv`. Este es análogo al `apply-env` del intérprete, pero opera sobre ambientes de tipos y emite errores específicos de tipado si la variable no se encuentra.

```scheme
(define type-of-expression
  (lambda (exp tenv)
    (cases expression exp
      (literal-expression (val)
        (if (number? val)
            (int-type)
            (bool-type)))
      (var-expression (var) (apply-tenv tenv var))
      ...
      )))
```

####
La regla de tipado para una expresión condicional (`if-expression`) establece tres condiciones. Primero, la expresión de prueba (`test-exp`) debe evaluarse a un tipo booleano (`bool-type`). Segundo, las expresiones de la rama verdadera (`true-exp`) y la rama falsa (`false-exp`) deben tener el mismo tipo. El tipo resultante de la expresión `if` completa será el tipo común de estas dos ramas.

Para implementar esto, se calculan recursivamente los tipos de las tres subexpresiones. Luego, se utilizan aserciones o funciones de verificación para confirmar que el tipo de la prueba es booleano y que los tipos de las dos ramas son iguales. Si se cumplen las condiciones, se retorna el tipo de la rama verdadera.

```scheme
(if-expression (test-exp true-exp false-exp)
  (let ((type-test (type-of-expression test-exp tenv))
        (type-true (type-of-expression true-exp tenv))
        (type-false (type-of-expression false-exp tenv)))
    (begin
      (check-equal-type type-test (bool-type) test-exp)
      (check-equal-type type-true type-false exp)
      type-true)))
```

####
La función auxiliar `check-equal-type` se utiliza para verificar que dos tipos son idénticos. Recibe tres argumentos: un tipo `t1`, un tipo `t2` y la expresión `exp` que se está evaluando. Si `t1` y `t2` no son iguales (`equal?`), la función genera un error de tipado, informando que los tipos no coinciden. El mensaje de error incluye la representación externa de ambos tipos y la expresión que causó el fallo, proporcionando contexto para la depuración. Si los tipos son iguales, la función no retorna valor y la ejecución continúa.

```scheme
(define check-equal-type
  (lambda (t1 t2 exp)
    (if (not (equal? t1 t2))
        (eopl:error 'check-equal-type
          "Types do not match: ~s and ~s in ~s"
          (type->external-form t1)
          (type->external-form t2)
          exp))))
```

####
Para mejorar la legibilidad de los mensajes de error de tipos, se utiliza una función de formato llamada `type->external-form`. Esta función toma una representación de tipo interna del sistema (como `atomic-type` o `proc-type`) y la convierte en una cadena de texto más comprensible para el programador. Por ejemplo, un `atomic-type` con la etiqueta `INT` se convierte en el símbolo `'int`. Un `proc-type` se formatea para mostrar los tipos de sus argumentos y su tipo de retorno, a menudo en una notación como `(T1 * T2 -> TResult)`. Esta conversión es crucial para que los errores de tipado sean informativos y fáciles de diagnosticar.
