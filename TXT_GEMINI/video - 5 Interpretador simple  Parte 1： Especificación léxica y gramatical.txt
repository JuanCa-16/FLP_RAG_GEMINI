El objetivo del curso es construir un interpretador simple que se irá desarrollando progresivamente. Se comenzará con un lenguaje capaz de realizar operaciones matemáticas básicas y se le añadirán características a lo largo del curso. El plan de estudio cubrirá procedimientos recursivos en el primer corte, seguido de tipos y, finalmente, objetos, abarcando así la totalidad del temario.

####

En el contexto de un lenguaje de programación, es crucial diferenciar entre valores expresados y valores denotados. Los valores expresados son aquellos que se manipulan y visualizan directamente en el código fuente del lenguaje, como un número literal. Por otro lado, los valores denotados son aquellos que están almacenados en la memoria, típicamente asociados a una variable. El primer lenguaje que se creará manejará exclusivamente números como valores.

####

La gramática inicial del lenguaje se define de la siguiente manera. Un programa consiste en una única expresión. A su vez, una expresión puede ser una de tres variantes: un número literal, un identificador (que representa una variable) o la aplicación de una operación primitiva. La aplicación de una primitiva se estructura con el nombre de la operación, seguido de una lista de expresiones entre paréntesis y separadas por comas. Toda la especificación del lenguaje se agrupa bajo una regla principal, en este caso, `programa`, para simplificar su procesamiento.

####

La especificación léxica define los componentes básicos o "tokens" del lenguaje.
1.  **Espacio en blanco**: Se utiliza la directiva `(space)` para identificar espacios, tabulaciones y saltos de línea, los cuales son ignorados por el analizador.
2.  **Comentarios**: Se definen como un carácter de porcentaje (`%`) seguido de cualquier secuencia de caracteres hasta un salto de línea. Los comentarios también son ignorados.
3.  **Identificadores**: Corresponden a una letra seguida de cualquier combinación de letras, dígitos o caracteres especiales. Se convierten en tokens de tipo `symbol`.
4.  **Números**: Se definen patrones para números enteros positivos, negativos y de punto flotante. Estos se convierten en tokens de tipo `number`. Por ejemplo, un entero es un dígito seguido de cero o más dígitos, y uno negativo comienza con un guion. Se añade también una regla para números flotantes que incluyen un punto decimal.

####

La especificación gramatical traduce la gramática abstracta en reglas concretas para el analizador sintáctico.
1.  `program`: Se define como una expresión (`expression`) y se le asigna la variante `a-program`.
2.  `expression`: Se definen sus variantes:
    *   Un `number` se mapea a `literal-expression`.
    *   Un `identifier` se mapea a `var-expression`.
    *   Una primitiva seguida de una lista de expresiones entre paréntesis se mapea a `prim-expression`. Para la lista de argumentos, se utiliza `(separated-list-of expression ",")` para indicar que son expresiones separadas por comas.
3.  `primitive`: Se definen las operaciones permitidas:
    *   `+` se mapea a `add-prim`.
    *   `-` se mapea a `minus-prim`.
    *   `*` se mapea a `mult-prim`.
    *   `add1` se mapea a `inc-prim` (incremento).
    *   `sub1` se mapea a `dec-prim` (decremento).
    *   Adicionalmente, se puede incluir `/` como `div-prim`.

####

Para crear el intérprete funcional, se utilizan las especificaciones léxica y gramatical para generar las estructuras de datos necesarias con la función `(make-define-datatypes)`. Esta función toma ambas especificaciones como entrada. Si hay errores, como un tipo de token no definido en la especificación léxica (por ejemplo, usar `identificador` en la gramática cuando se definió `identifier` en la léxica), el sistema lo notificará. Tras una ejecución exitosa, se generan los tipos de datos correspondientes a la gramática.

####

El ciclo de lectura, evaluación e impresión (REPL) se construye con la función `(make-read-eval-print-loop)`. Esta función requiere tres componentes:
1.  Un `prompt` para la interfaz de usuario (ej: `'> '`).
2.  Una función de evaluación, que se llamará `eval-program`. Esta función recibirá el programa analizado (el árbol de sintaxis abstracta) y se encargará de interpretarlo.
3.  Un `parser`, creado con `(make-string-parser)`, que a su vez recibe las especificaciones léxica y gramatical.
Finalmente, se define la función `eval-program`, que inicialmente solo mostrará la estructura de datos del programa analizado para verificar que el parsing es correcto.

####

Una vez completadas las especificaciones léxica y gramatical y construido el REPL, el sistema es capaz de analizar el código de entrada y generar el árbol de sintaxis abstracta. Por ejemplo, al introducir la expresión `(+ 5,4)`, el intérprete la analiza correctamente y muestra la estructura de datos correspondiente a una aplicación de la primitiva de suma con los literales 5 y 4 como argumentos. Este resultado confirma que el analizador sintáctico funciona según lo especificado. El siguiente paso consiste en implementar la lógica de evaluación dentro de la función `eval-program` para interpretar estas estructuras y producir un resultado.
