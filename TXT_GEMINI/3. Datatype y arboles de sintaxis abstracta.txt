Dos tipos no pueden tener el mismo nombre; tampoco dos variantes, aunque pertenezcan a tipos diferentes, pueden tener el mismo nombre. Para cada variante, un procedimiento constructor es creado. Si hay `n` campos en una variante, su constructor recibe `n` argumentos, prueba si cada uno de ellos satisface el predicado asociado y retorna un nuevo valor de dicha variante del tipo de dato. El nombre `nombre-predicado-tipo` es ligado a un predicado. Este predicado determina si su argumento es un valor perteneciente al tipo `nombre-tipo`.

####
<CODIGO_BLOCK>
A continuación, se define el tipo de dato `lc-exp` para representar expresiones del cálculo lambda. Este tipo tiene tres variantes: `var-exp` para variables, `lambda-exp` para abstracciones (funciones) y `app-exp` para aplicaciones (llamadas a función).

```racket
(define-datatype
  lc-exp
  lc-exp?
  (var-exp
    (id identifier?))
  (lambda-exp
    (id identifier?)
    (body lc-exp?))
  (app-exp
    (rator lc-exp?)
    (rand lc-exp?)))
```
</CODIGO_BLOCK>

<CODIGO_BLOCK>
Se definen los tipos de dato para listas de S-expressions (`s-list`) y para S-expressions (`s-exp`). Una `s-list` puede ser vacía o no vacía, conteniendo un primer elemento y el resto de la lista. Una `s-exp` puede ser un símbolo o una lista.

```racket
(define-datatype
  s-list
  s-list?
  (empty-s-list)
  (non-empty-s-list
    (first s-exp?)
    (rest s-list?)))

(define-datatype
  s-exp s-exp?
  (symbol-s-exp
    (sym symbol?))
  (s-list-s-exp
    (slst s-list?)))
```
</CODIGO_BLOCK>

####
<CODIGO_BLOCK>
Esta es una definición alternativa para el tipo de dato `s-list`. En esta versión, se utiliza un procedimiento auxiliar `list-of` para definir una lista de `s-exp`. El procedimiento `list-of` es un constructor de predicados que verifica si un valor es una lista donde cada elemento satisface un predicado dado.

```racket
(define-datatype
  s-list
  s-list?
  (an-s-list
    (sexps (list-of s-exp?))))

(define list-of
  (lambda (pred)
    (lambda (val)
      (or (null? val)
          (and (pair? val)
               (pred (car val))
               ((list-of pred) (cdr val)))))))
```
</CODIGO_BLOCK>

####
Para determinar a qué variante de un tipo de dato pertenece un objeto y extraer sus componentes, se usa la forma `cases`, la cual tiene la sintaxis general:

```racket
(cases nombre-tipo expresion
  [(nombre-variante ({nombre-campo}*) consecuente)]*
  [else por-defecto])
```

La expresión `cases` evalúa `expresion`. Esto da como resultado un valor `v` de tipo `nombre-tipo`. Si `v` es una variante `nombre-variante`, cada uno de los campos `nombre-campo` son asociados al valor del correspondiente campo de `v`. Luego la expresión `consecuente` es evaluada y su valor es retornado. Si `v` no es una de las variantes y la cláusula `else` es especificada, la expresión `por-defecto` es evaluada y su valor retornado. Si no existe una cláusula `else`, entonces tiene que existir una cláusula para cada una de las variantes del tipo de dato.

####
<CODIGO_BLOCK>
El procedimiento `occurs-free?` determina si una variable (`search-var`) aparece libre en una expresión del cálculo lambda (`exp`). Utiliza `cases` para deconstruir la expresión y aplicar la lógica correspondiente a cada tipo de variante.

```racket
(define occurs-free?
  (lambda (search-var exp)
    (cases lc-exp exp
      (var-exp (var)
        (eqv? search-var var))
      (lambda-exp (bound-var body)
        (and
          (not (eqv? search-var bound-var))
          (occurs-free? search-var body)))
      (app-exp (rator rand)
        (or (occurs-free? search-var rator)
            (occurs-free? search-var rand))))))
```
</CODIGO_BLOCK>

<CODIGO_BLOCK>
Definición del tipo de dato `bintree` para representar árboles binarios. Un árbol puede ser una hoja (`leaf-node`), que contiene un número, o un nodo interior (`interior-node`), que contiene una clave simbólica y dos subárboles (izquierdo y derecho).

```racket
(define-datatype
  bintree
  bintree?
  (leaf-node
    (datum number?))
  (interior-node
    (key symbol?)
    (left bintree?)
    (right bintree?)))
```
</CODIGO_BLOCK>

####
A continuación, se define un procedimiento que permite encontrar la suma de los enteros en las hojas de un árbol. Usando `cases` se tiene:
<CODIGO_BLOCK>
El procedimiento `leaf-sum` calcula la suma de todos los valores numéricos (`datum`) en los nodos hoja de un árbol binario (`bintree`). Recorre el árbol de forma recursiva, sumando los valores de las hojas y combinando los resultados de los subárboles.

```racket
(define leaf-sum
  (lambda (tree)
    (cases bintree tree
      (leaf-node (datum)
        datum)
      (interior-node (key left right)
        (+ (leaf-sum left)
           (leaf-sum right))))))
```
</CODIGO_BLOCK>

<CODIGO_BLOCK>
Definición de un tipo de dato para representar ambientes (entornos de ejecución). Un ambiente puede ser vacío (`empty-env-record`) o extendido (`extended-env-record`), que contiene listas de símbolos y sus valores asociados, además de una referencia a un ambiente anterior. El predicado `scheme-value?` se define para aceptar cualquier valor.

```racket
(define-datatype
  environment
  environment?
  (empty-env-record)
  (extended-env-record
    (syms (list-of symbol?))
    (vals (list-of scheme-value?))
    (env environment?)))

(define scheme-value?
  (lambda (v) #t))
```
</CODIGO_BLOCK>

####
<CODIGO_BLOCK>
El procedimiento `apply-env` busca el valor asociado a un símbolo (`sym`) dentro de un ambiente (`env`). Si el ambiente está vacío, reporta un error. Si es un ambiente extendido, busca el símbolo en la lista de símbolos; si lo encuentra, retorna el valor correspondiente, y si no, continúa la búsqueda en el ambiente anterior.

```racket
(define apply-env
  (lambda (env sym)
    (cases environment env
      (empty-env-record ()
        (eopl:error 'apply-env "No binding for ~s" sym))
      (extended-env-record (syms vals old-env)
        (let ((pos (list-find-position sym syms)))
          (if (number? pos)
            (list-ref vals pos)
            (apply-env old-env sym)))))))
```
</CODIGO_BLOCK>

####
Dada la gramática de las expresiones del cálculo lambda:
⟨expresión⟩ ::= ⟨identificador⟩
⟨expresión⟩ ::= (`lambda` (`⟨identificador⟩`) `⟨expresión⟩`)
⟨expresión⟩ ::= (`⟨expresión⟩` `⟨expresión⟩`)

Se puede representar cada expresión del cálculo lambda usando el tipo de dato `lc-exp` definido anteriormente mediante `define-datatype`.

####
Una BNF especifica una representación particular de un tipo de dato que usa los valores generados por la gramática. Esta representación es llamada sintaxis concreta o representación externa. Para procesar dichos datos, se requiere convertirlos a una representación interna o sintaxis abstracta, en la cual los símbolos terminales (como paréntesis) no necesiten ser almacenados ya que no llevan información.

####
Para crear una sintaxis abstracta a partir de una sintaxis concreta, se debe nombrar cada regla de producción de la sintaxis concreta y cada ocurrencia de un símbolo no terminal. Para la gramática de las expresiones del cálculo λ, se puede resumir la relación entre la sintaxis concreta y la abstracta usando la siguiente notación:

*   `⟨expresión⟩`
    *   `⟨identificador⟩` se convierte en `var-exp (id)`
    *   `(` `lambda` `(` `⟨identificador⟩` `)` `⟨expresión⟩` `)` se convierte en `lambda-exp (id body)`
    *   `(` `⟨expresión⟩` `⟨expresión⟩` `)` se convierte en `app-exp (rator rand)`

####
La sintaxis abstracta de una expresión es más fácil de comprender visualizándola como un árbol de sintaxis abstracta.
<IMG_BLOCK>
Descripción del árbol de sintaxis abstracta para la expresión `(lambda (x) (f (f x)))`: La raíz es un nodo `lambda-exp` con `x` como identificador y un subárbol como cuerpo. El cuerpo es un nodo `app-exp` cuyo operador (`rator`) es el nodo `var-exp` para `f`, y cuyo operando (`rand`) es otro subárbol `app-exp`. Este último `app-exp` tiene como operador a `f` y como operando al nodo `var-exp` para `x`.
</IMG_BLOCK>
Los árboles de sintaxis son útiles en lenguajes de programación de procesamiento de sistemas, ya que los programas que procesan otros programas (como los interpretadores o compiladores) son casi siempre dirigidos por sintaxis. Esto es, que el procesamiento de cada parte de un programa es guiado por la regla gramatical asociada con dicha parte, y cualquier subparte correspondiente a un símbolo no terminal puede ser accedida con facilidad.

####
Cuando se procesa la expresión `(lambda (x) (f (f x)))`, primero se debe reconocer como una expresión del cálculo lambda, correspondiente a la regla:
⟨expresión⟩ ::= (`lambda` (`⟨identificador⟩`) `⟨expresión⟩`)
El parámetro formal es `x` y el cuerpo es `(f (f x))`. El cuerpo debe ser reconocido como una `app-exp`, y así sucesivamente.

####
El problema de convertir un árbol de sintaxis abstracta a una representación de listas y símbolos (con la cual Scheme mostraría las expresiones en su sintaxis concreta), se resuelve con el siguiente procedimiento:
<CODIGO_BLOCK>
La función `unparse-expression` toma un árbol de sintaxis abstracta (`exp` del tipo `lc-exp`) y lo convierte de nuevo a su representación como S-expression de Racket. Para cada variante del tipo de dato, construye la lista correspondiente.

```racket
(define unparse-expression
  (lambda (exp)
    (cases lc-exp exp
      (var-exp (id)
        id)
      (lambda-exp (id body)
        (list 'lambda (list id) (unparse-expression body)))
      (app-exp (rator rand)
        (list
          (unparse-expression rator)
          (unparse-expression rand))))))
```
</CODIGO_BLOCK>

####
La tarea de derivar el árbol de sintaxis abstracta a partir de una cadena de caracteres es denominado *parsing*, y es llevado a cabo por un programa llamado *parser* (analizador sintáctico). El siguiente procedimiento deriva la representación en sintaxis concreta (como S-expression) a árboles de sintaxis abstracta.
<CODIGO_BLOCK>
El procedimiento `parse-expression` funciona como un analizador sintáctico que convierte una S-expression de Racket (`datum`) en su correspondiente árbol de sintaxis abstracta (`lc-exp`). Distingue entre símbolos (variables), listas que comienzan con `lambda` (abstracciones) y otras listas (aplicaciones).

```racket
(define parse-expression
  (lambda (datum)
    (cond
      ((symbol? datum)
       (var-exp datum))
      ((pair? datum)
       (if (eqv? (car datum) 'lambda)
         (lambda-exp
           (caadr datum)
           (parse-expression (caddr datum)))
         (app-exp
           (parse-expression (car datum))
           (parse-expression (cadr datum)))))
      (else
       (eopl:error
         'parse-expression
         "Invalid concrete syntax ~s" datum)))))
```
</CODIGO_BLOCK>
